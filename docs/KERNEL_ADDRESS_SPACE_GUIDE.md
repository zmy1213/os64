# 从第一版 `TSS` 到“每进程地址空间骨架”

这一轮没有直接跳到：

```text
ring 3 用户程序已经跑起来
```

而是先补中间一层非常关键、又非常容易被跳过的基础设施：

> 每进程第一次真正带上一份自己的页表根（address space root）

一句话先说结论：

```text
以前：所有代码默认都只活在“当前这一个 CR3”里
现在：已经能克隆当前页表根、给新地址空间额外挂用户页，并把这份地址空间挂进 PCB
```

这一步的意义是：

> 把“只有一个全局内核地址空间的教学内核”推进成“进程对象已经知道自己的页表根是谁”的系统。

---

## 1. 为什么这一轮先做地址空间，而不是直接做 `iretq`

如果你现在直接冲去做：

- ring 3 入口帧
- `iretq`
- 用户栈
- 用户态打 `int 0x80`

你会马上遇到一个问题：

```text
那个“用户进程”到底归哪一份页表管？
```

如果这个问题还没先解决，
那你后面做的很多东西都会漂在空中：

- 用户栈放哪
- 用户代码页放哪
- 以后 `fork/exec` 复制和切换的是哪份页表
- 内核切回用户态时切的是哪一个 CR3

所以在 `TSS` 之后，
最合理的下一步不是立刻做“真正切用户态”，
而是先补：

```text
AddressSpace
```

也就是：

> 先把“一个进程对应一份页表根”这个形状立起来。

---

## 2. 这一轮到底新增了什么

主要补了 3 层：

### 2.1 `paging` 不再只会改“当前 CR3”

以前 `map_page()` 的含义其实是：

```text
往当前正在运行的那份页表里挂一张页
```

这对早期单地址空间内核够用，
但一旦你要为“另一个进程的页表”提前准备映射，
就不够了。

所以这一轮把它扩成：

- `paging_current_root_physical()`
- `map_page_in_root(...)`
- `resolve_physical_address_in_root(...)`

也就是说现在不仅能改：

```text
当前 CR3
```

也能改：

```text
某一份指定的页表根
```

这一步非常关键，
因为以后：

- 加载用户程序
- 建用户栈
- 准备 `argv/envp`

这些工作很多都要在“还没切过去运行之前”先写到目标地址空间里。

---

### 2.2 新增 `memory/address_space.*`

这一层不是“又造一套页表代码”，
而是把“进程视角的地址空间”单独包出来。

现在的 `AddressSpace` 至少会记住：

- `root_physical_address`
- `root_virtual_address`
- `owns_page_table_root`
- `user_region_base`
- `user_region_limit`
- `default_user_stack_top`
- `mapped_user_pages`

你可以把它理解成：

```text
Page table = CPU 真正执行时看的底层结构
AddressSpace = 内核里拿来管理“这份页表属于谁、准备怎么用”的高层对象
```

---

### 2.3 `ProcessControlBlock` 第一次真的带上地址空间

以前 PCB 里主要是：

- pid
- state
- live_thread_count
- total_thread_ticks

这些更偏“调度/生命周期”。

现在又多了一项：

```text
address_space
```

这就意味着：

> 进程对象不再只是“线程的拥有者”，而开始携带“这条执行流未来应该跑在哪份页表里”。

虽然现在还没有真正的用户线程，
但这个形状已经和现代内核更接近了。

---

## 3. 为什么不是“从零手写一份全空页表”，而是先克隆当前 CR3

这是这一轮最容易让人疑惑的一点。

很多人会问：

```text
既然是新进程，
为什么不直接给它一份全新的空页表？
```

原因是当前项目还没做到：

- 高半区内核
- 用户低地址 / 内核高地址的严格隔离
- 内核全局映射模板

现在内核本身还活在低地址的恒等映射里，
比如：

```text
0 ~ 2 MiB
```

这里面就有：

- 启动期页表
- 内核代码
- 一些早期数据结构

所以如果你现在直接给新地址空间一份“完全空白”的页表，
然后将来真的切过去，
CPU 连内核当前这套最基本的执行环境都找不到。

因此这一轮采取的是一个非常现实的教学方案：

> 先把当前 CR3 深拷贝成一份新的页表根，再在这份副本上增量添加“将来的用户页”。

这样做的好处是：

1. 新地址空间先天就保留了当前内核能跑起来所需的基础映射
2. 后面再额外挂用户页，不会直接改坏当前活着的那份页表
3. 小白更容易理解“同一套基础内核映射 + 每进程私有补丁页”这个过渡形状

---

## 4. “深拷贝页表根”到底拷了什么

不是只拷最顶上的 PML4。

如果你只拷：

```text
PML4 那一页
```

但下面 PDPT / PD / PT 还是和当前内核共用，
那你后面往“新地址空间”里加一张页时，
很可能还是会把当前地址空间那一份下级表改掉。

所以这一轮的做法是：

```text
递归克隆现有页表树
```

规则是：

- 遇到不存在项：直接留空
- 遇到普通“指向下一层页表”的项：继续递归克隆下一层
- 遇到大页项：直接复制这条映射

这样最终得到的是：

```text
新的 PML4
+ 新的 PDPT
+ 新的 PD / PT（如果存在）
+ 保留原来的物理页映射目标
```

也就是说：

```text
“映射关系相同”
不等于
“页表结构还共用”
```

这正是这一步想先建立起来的概念。

---

## 5. 这一轮“用户区”先放在哪

现在代码里约定了一个很小、很好记的窗口：

```text
user base  = 0x0000000000400000   // 4 MiB
user limit = 0x0000000000800000   // 8 MiB
stack top  = 0x0000000000800000
```

先强调一下：

这不是最终现代内核的地址布局，
只是这一轮教学里故意选的一个：

- 好记
- 和当前 0~2 MiB 恒等映射错开
- 方便做第一版 smoke test

也就是说现在先不是追求：

```text
完美地址空间设计
```

而是先追求：

```text
“新地址空间里已经能有一段明确的、独立的用户页区域”
```

---

## 6. 当前实现还不算“真正隔离”的原因是什么

这里要非常诚实地讲清楚：

> 这一轮只是地址空间骨架，不是安全隔离完成版。

原因在于当前内核还没有做到：

- 内核搬到高半区
- 用户态不能直接碰内核同一段低地址
- 每进程独立的内核栈 + 用户栈切换路径
- 切 CR3 + `iretq` 进入 ring 3

所以现在这一步的真实定位是：

```text
先把“每进程有自己的页表根”这件事立住
```

而不是：

```text
已经完成用户/内核隔离
```

这两者差很多，
一定不要混在一起。

## 7. 这一轮改了哪些文件

主要是这些：

- `kernel/memory/paging.hpp`
- `kernel/memory/paging.cpp`
- `kernel/memory/address_space.hpp`
- `kernel/memory/address_space.cpp`
- `kernel/task/scheduler.hpp`
- `kernel/task/scheduler.cpp`
- `kernel/core/kernel_main.cpp`
- `scripts/test-stage1.sh`
- `scripts/test-invalid-opcode.sh`
- `scripts/test-page-fault.sh`

一句话分工：

- `paging.*`
  负责“给指定页表根挂页 / 查映射”。
- `address_space.*`
  负责“克隆当前地址空间并管理用户区布局”。
- `scheduler.*`
  负责让 PCB 真正携带 `AddressSpace`。
- `kernel_main.cpp`
  负责跑第一版地址空间烟测。
- `scripts/test-*.sh`
  负责把这条新主线纳入自动回归。

## 8. 启动日志里现在会多看到什么

正常启动链里现在会多出这些日志：

- `address_space_kernel_root=0x...`
- `address_space_user_root=0x...`
- `address_space_user_base=0x0000000000400000`
- `address_space_user_stack_top=0x0000000000800000`
- `address_space_user_page_phys=0x...`
- `address_space_user_page_virt=0x0000000000400000`
- `address_space_user_lookup=0x...`
- `address_space_mapped_user_pages=1`
- `address_space ok`

你可以这样理解：

- `kernel_root`
  当前内核正在使用的那份 CR3 根。
- `user_root`
  新克隆出来的那份地址空间根。
- `user_base / user_stack_top`
  这一步先约定好的“将来用户区从哪到哪”。
- `user_page_phys`
  给新地址空间准备的一张物理页。
- `user_page_virt`
  把它挂到新地址空间里的哪个虚拟地址。
- `user_lookup`
  不切 CR3 也能从页表里把这条映射查回来。
- `mapped_user_pages=1`
  说明这个新地址空间里已经有 1 张额外用户页。

---

## 9. 现在这个烟测到底证明了什么

它至少证明了 4 件事：

1. 当前 CR3 已经能被读出来
2. 当前地址空间已经能被深拷贝成一份新的根
3. 新地址空间可以在用户区额外挂页
4. 这条映射可以在不切换执行上下文的情况下被页表遍历逻辑正确解析出来

注意它没有证明的是：

- 真正切进这份地址空间执行
- 用户代码能在里面跑
- ring 3 / ring 0 切换正确

这些是后面的事。

---

## 10. 如果你现在是小白，最该抓住哪句话

这一轮最重要的理解是：

> `TSS` 解决的是“CPU 进内核时切哪根栈”，`AddressSpace` 解决的是“这个进程未来该活在哪份页表里”。

这两件事都和“进入用户态”有关，
但它们不是一回事。

你只要把这个区分记住，
后面再看：

- CR3 切换
- 用户栈
- `iretq`
- `int 0x80`

就不会把几层概念揉成一团。

## 11. 怎么测试这一轮

顺序还是保持不变：

```bash
make test-stage1
make test-invalid-opcode
make test-page-fault
```

原因和前面一样：

```text
它们共用 build/disk.img
```

而且现在三条测试都会顺便覆盖到：

```text
TSS + address_space + trap
```

这一整段更靠前的初始化主线。

---

## 12. 下一步最合理做什么

现在最合理的下一步，
终于可以开始碰“真正用户态”了，
但还是建议按下面顺序推进：

1. 先为用户线程准备最小入口栈帧
2. 给线程对象区分“内核线程入口”和“用户线程入口”
3. 用 `iretq` 第一次真正进入 ring 3
4. 让已有的 `int 0x80` 从 ring 3 打回 ring 0

一句话说：

> 现在“切哪根内核栈”有了，`TSS`；“活在哪份页表”有了，`AddressSpace`；下一步才轮到“真的切进用户态跑起来”。
