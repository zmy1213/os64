# 从最小 wait / sleep 到键盘 IRQ

这份文档专门讲这一轮为什么继续做：

- 键盘 IRQ1
- 键盘扫描码
- 最小键盘中断处理
- 不用手工按键的自动化自测

---

## 1. 为什么这一步是自然的下一步

上一轮你已经有了：

- PIT 定时器中断
- 全局 tick
- `timer_wait_ticks()`
- `timer_sleep_ms()`

也就是说，
你的内核已经会：

> 按时间醒来。

但现代内核不能只会对“时间流逝”有反应，
它还必须开始对：

- 用户输入
- 设备完成事件
- 外部世界真正发生的事情

有反应。

而这类输入设备里，
最经典、最适合做第一步的就是：

> 键盘。

所以这一轮最自然的目标就是：

1. 放开键盘对应的 IRQ1
2. 收到一次真正的键盘中断
3. 从控制器里读出 1 个扫描码

---

## 2. 键盘 IRQ1 是什么

在传统 PC 架构里，
键盘通常走的是：

- 主 PIC 的 IRQ1

前一轮我们已经有了：

- IRQ0 = PIT 定时器

这一轮继续把：

- IRQ1 = 键盘

也接进来。

因为 PIC 之前已经被重映射到 `32~47`，
所以现在：

- IRQ0 -> 向量 `32`
- IRQ1 -> 向量 `33`

这就是为什么这轮 `kernel_handle_irq()` 里，
会继续识别：

- `vector == 33`

---

## 3. 什么是扫描码

小白版可以这样理解：

> 扫描码不是字符，而是键盘硬件发出来的“按键编号”。

比如你按下 `A`，
键盘通常不会直接告诉内核：

- “这是字母 A”

它先发的是一个更底层的值，
例如这一轮测试里用的：

- `0x1E`

这是经典 Set 1 扫描码里 `A` 键按下时的 make code。

所以这一轮你看到内核记录的是：

- `keyboard_last_scancode=0x...1E`

而不是直接显示字符 `A`。

---

## 4. 这一轮代码具体做了什么

这一轮主要加了一个新的模块：

- `kernel/interrupts/keyboard.cpp`
- `kernel/interrupts/keyboard.hpp`

它先只做最小的 4 件事：

### 1. 初始化键盘状态

`initialize_keyboard()` 会先：

- 清掉控制器输出缓冲区里可能残留的旧字节
- 把 IRQ 计数器清零
- 把最近扫描码清零

为什么要先清旧数据？

因为 BIOS 或更早的阶段，
有可能已经往键盘控制器里留过旧内容。

如果不清，
你后面第一次收到的“键盘数据”可能根本不是你当前测试想看的那一个。

### 2. 收到 IRQ1 时读取扫描码

`handle_keyboard_irq()` 做的事情很简单：

1. 看控制器输出缓冲区是不是有数据
2. 如果有，就从端口 `0x60` 读 1 个字节
3. 把这个字节记到 `g_keyboard_last_scancode`
4. 把 `g_keyboard_irq_count` 加 1

这就是第一版最小键盘中断处理。

### 3. PIC 不再只会固定放开 IRQ0

前一轮 `initialize_pic()` 之后，
PIC 只是默认：

- 放开 IRQ0
- 其他都先关着

这一轮继续加了：

- `enable_pic_irq(uint8_t irq_line)`
- `disable_pic_irq(uint8_t irq_line)`

这样现在内核就能在后续阶段按需放开：

- IRQ1 键盘
- 以后别的设备 IRQ

这一步的意义很大，
因为内核开始从：

- “写死只接一个中断”

变成：

- “能按设备逐步开放中断线”

### 4. IRQ 分发里新增键盘路径

现在 `kernel_handle_irq()` 不再只看：

- `vector == 32`

还会继续识别：

- `vector == 33`

也就是：

- 定时器走 `handle_timer_irq()`
- 键盘走 `handle_keyboard_irq()`

---

## 5. 为什么这轮测试不用你手工按键

如果完全靠真实按键做测试，
自动化就会很麻烦。

因为脚本跑 QEMU 时，
不一定有人正好在窗口里按下某个键。

所以这一轮用了一个很实用的办法：

> 让键盘控制器自己“假装收到了一次键盘输入”。

具体做法是往 `0x64` 写：

- `0xD2`

这是一条经典 8042 控制器命令，
意思可以理解成：

> “把我接下来写的那个字节，塞进输出缓冲区，假装它是键盘刚送来的数据。”

然后再往 `0x60` 写：

- `0x1E`

于是控制器就会像真的键盘刚发来了 `A` 的扫描码一样，
把这个字节挂进输出缓冲区，并触发 IRQ1。

这一招的好处是：

1. 不需要人手工干预
2. 测试能自动跑
3. 还能完整走一遍真正的 IRQ1 处理路径

---

## 6. `kernel_main()` 里的键盘自测在验证什么

这一轮 `kernel_main.cpp` 里新增了一段：

- `run_keyboard_smoke_test()`

它大致按这个流程跑：

1. 初始化键盘状态
2. 通过 `enable_pic_irq(1)` 放开 IRQ1
3. 打开中断
4. 注入一个测试扫描码 `0x1E`
5. 等待 IRQ1 到来
6. 读取 IRQ 计数和最后扫描码
7. 检查它们是否符合预期

预期日志会看到：

- `keyboard init ok`
- `keyboard irq1 enabled`
- `keyboard_test_scancode=0x000000000000001E`
- `keyboard_irq_count=1`
- `keyboard_last_scancode=0x000000000000001E`
- `keyboard ok`

这 6 行分别说明：

1. 键盘状态清理成功
2. IRQ1 已经真的放开
3. 注入的测试数据是什么
4. 键盘 IRQ 真的来过 1 次
5. 收到的数据和注入的一样
6. 整条路径总体通过

---

## 7. 这一轮和上一轮时间系统是什么关系

这两轮其实是连着的。

上一轮你做的是：

- 没事先 `hlt`
- 等 tick 把 CPU 唤醒

这一轮键盘测试继续复用了这个思路：

- 如果键盘 IRQ 还没来
- 就继续 `hlt`
- 期间 timer IRQ 会不断把 CPU 周期性唤醒
- 顺便检查是否已经收到键盘 IRQ

也就是说：

> timer 现在不仅能用来睡眠，还能给别的事件等待提供“超时心跳”。

这一步虽然还很小，
但已经非常像真正内核里常见的模式了。

---

## 8. 怎么测试这一轮

命令：

```bash
make test-stage1
```

预期日志除了前面的 timer 输出外，
还会新增：

- `keyboard init ok`
- `keyboard irq1 enabled`
- `keyboard_test_scancode=0x000000000000001E`
- `keyboard_irq_count=1`
- `keyboard_last_scancode=0x000000000000001E`
- `keyboard ok`

另外这两条异常测试也应该继续通过：

```bash
make test-invalid-opcode
make test-page-fault
```

因为这能证明：

- 新增键盘 IRQ 路径之后
- 原来的异常链没有被破坏

## 9. 这一轮为什么顺手暴露出了 stage2 的读盘 bug

这一轮还有一个很值得学习的点：

> 新功能有时不是“自己坏了”，而是把更底层早就潜伏的 bug 逼出来了。

现象是这样的：

1. 键盘 IRQ 代码加进来以后，`kernel.bin` 体积变大了
2. 测试时 `keyboard` 相关日志有些正常，有些却突然乱码
3. 比如前面的字符串还对，后面某些 `reserved`、`idt ok`、`boot info ok` 却变成了脏数据

这说明什么？

这通常不是“键盘中断把字符串打坏了”，
而更像是：

- 内核前半段被正确加载了
- 内核后半段没有被正确加载

后来定位到真因是：

- stage2 的 CHS 读盘代码在跨柱面时把 `CH` 里的 `cylinder` 覆盖掉了
- 所以前 `27` 个扇区正常
- 从第 `28` 个扇区开始读错

为什么键盘这一轮最容易把它暴露出来？

因为这轮新增的代码和字符串，
正好把 `kernel.bin` 推过了“跨柱面”这个边界。

这对你学内核很重要，
因为它说明了一件真实的工程经验：

> 一个新模块把系统搞坏，不一定是新模块本身写错了，也可能只是它让旧的边界 bug 第一次真正出现。

---

## 10. 这一轮完成后，你真正多了什么能力

现在这个内核已经不只是：

- 会收到时钟中断
- 会按 tick 睡眠

而是继续多了：

1. 第一次接收输入设备 IRQ
2. 第一次读取真实外设风格的数据字节
3. 第一次把“时间事件”和“输入事件”同时纳入内核事件系统

这一步意义很大，
因为它标志着你的内核开始从：

- 只会处理自己内部时间逻辑

走向：

- 会处理外部世界送进来的事件

---

## 11. 这一版的局限也要看清楚

现在它仍然只是最小键盘 IRQ 版本，
所以还有这些限制：

1. 只记录扫描码，还没有真正翻译成字符
2. 还没有区分按下和松开做更完整状态管理
3. 还没有键盘缓冲区或事件队列
4. 还没有面向 shell/终端的输入接口
5. 当前注入测试依赖 8042/QEMU 这类兼容环境

这些都正常，
因为这一轮目标只是：

> 先把“键盘 IRQ1 -> 读出扫描码”这条主链打通。

---

## 12. 下一步最合理做什么

有了键盘 IRQ 之后，
下一步最自然的通常是下面几条路之一：

1. 做扫描码到字符的第一版翻译
2. 做键盘环形缓冲区
3. 做最小 `getc()` / 控制台输入
4. 再往上做一个最小命令行

如果按“最适合继续学懂”的顺序，
我建议下一步优先做：

> 键盘缓冲区 + 扫描码到字符的最小翻译。

一句话总结这一轮：

> 现在内核不只是会被“时间”唤醒，也第一次会被“用户输入事件”真正打断并处理了。
