# Stage2 进入保护模式说明

这份文档讲的是第三轮真正做了什么：

> 在 `stage2` 里开启 A20，读取 BIOS `E820` 内存映射，建立 `GDT`，并切进 32 位保护模式。

这不是“为了显得更高级”，而是下一条启动链必须跨过的门槛。

---

## 1. 为什么这一步必须放在 stage2

`stage1` 的空间只有 512 字节。

它适合做的事只有：

- 建最小环境
- 打印最早期日志
- 读磁盘
- 跳到更大的加载器

而下面这些事都更复杂：

- A20 检查和开启
- BIOS `E820` 枚举
- `GDT` 构建
- 切保护模式

所以这些逻辑必须进入 `stage2`。

---

## 2. 为什么先开 A20

如果 A20 没开，CPU 在某些地址访问上仍然会发生 1MiB 回绕。

后面你一旦要：

- 访问高地址
- 建更正式的加载器环境
- 准备分页和 long mode

不开 A20 就会埋下隐患。

所以顺序上先做：

1. 检查 A20 是否已开
2. 如果没开，就通过 `0x92` 端口开启
3. 再做一次检查确认结果

为什么这么做：

- 先检查，避免无意义改状态
- 端口 `0x92` 是当前最简单、在 QEMU 上最稳定的 fast A20 方案
- 再检查一次，避免把“我写了代码”和“它真的生效了”混为一谈

---

## 3. 为什么接着读 E820

光能启动还不够，内核以后必须知道：

- 哪些内存能用
- 哪些内存不能碰

而 BIOS `E820` 正是实模式下最标准的内存布局接口之一。

为什么现在就做：

- 现在你还在实模式，BIOS 中断还能直接用
- 进入保护模式后，再直接用 BIOS 服务会麻烦很多

所以这一步要放在切模式之前做。

---

## 4. 为什么再建 GDT

保护模式不是“写一条指令就切进去”。

它至少要先有一套合法的段描述符，也就是 `GDT`。

这里先用最小三项：

- null descriptor
- 32 位 code segment
- 32 位 data segment

为什么这么做：

- 现在目标只是先成功进入 32 位保护模式
- 不是马上做复杂分段系统
- 所以用最小平坦模型最稳

---

## 5. 为什么最后才进入保护模式

因为保护模式之后，最方便的 BIOS 实模式服务就不再能直接随手用了。

所以顺序必须是：

1. 先做所有依赖 BIOS 的事情
2. 再切模式

这也是为什么这轮严格按下面顺序来：

1. `stage2 ok`
2. `a20 ok`
3. `e820 ok`
4. `protected mode ok`

每个里程碑都能单独验证问题位置。

---

## 6. 这轮实现后，启动链推进到了哪里

这一轮如果成功，你得到的不是一个完整内核，而是更强的启动链：

```text
BIOS
-> stage1
-> stage2
-> A20 enabled
-> E820 memory map collected
-> GDT loaded
-> protected mode
```

这一步的意义非常大，因为它说明你已经脱离“只有实模式 boot sector”的阶段了。

---

## 7. 下一步为什么不是马上 long mode

long mode 确实是目标，但现在直接跳过去会把问题堆太多。

在保护模式和 long mode 之间，你最好先补这些：

1. 更稳定的 32 位打印与调试
2. 更清楚的内存映射数据结构
3. 内核加载地址规划
4. 页表准备

原因是：

> long mode 的前置条件比保护模式更多，如果中间层不稳，很容易一黑屏就完全失去定位能力。

所以这一步先把保护模式站稳，是更合理的做法。

---

## 8. 这一步后来继续推进成了什么

后面的实现已经继续往下做了，不再只停在保护模式。

现在后续已经补上了：

1. 最小页表
2. `PAE`
3. `EFER.LME`
4. 打开分页
5. 跳进 64 位代码段

也就是说，启动链后来已经从：

`... -> protected mode`

继续推进成：

`... -> protected mode -> paging -> long mode`
