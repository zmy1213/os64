# Stage2 进入保护模式说明

这份文档讲的是第三轮真正做了什么：

> 在 `stage2` 里开启 A20，读取 BIOS `E820` 内存映射，建立 `GDT`，并切进 32 位保护模式。

这不是“为了显得更高级”，而是下一条启动链必须跨过的门槛。

---

## 1. 为什么这一步必须放在 stage2

`stage1` 的空间只有 512 字节。

它适合做的事只有：

- 建最小环境
- 打印最早期日志
- 读磁盘
- 跳到更大的加载器

而下面这些事都更复杂：

- A20 检查和开启
- BIOS `E820` 枚举
- `GDT` 构建
- 切保护模式

所以这些逻辑必须进入 `stage2`。

---

## 2. 为什么先开 A20

如果 A20 没开，CPU 在某些地址访问上仍然会发生 1MiB 回绕。

后面你一旦要：

- 访问高地址
- 建更正式的加载器环境
- 准备分页和 long mode

不开 A20 就会埋下隐患。

所以顺序上先做：

1. 检查 A20 是否已开
2. 如果没开，就通过 `0x92` 端口开启
3. 再做一次检查确认结果

为什么这么做：

- 先检查，避免无意义改状态
- 端口 `0x92` 是当前最简单、在 QEMU 上最稳定的 fast A20 方案
- 再检查一次，避免把“我写了代码”和“它真的生效了”混为一谈

---

## 3. 为什么接着读 E820

光能启动还不够，内核以后必须知道：

- 哪些内存能用
- 哪些内存不能碰

而 BIOS `E820` 正是实模式下最标准的内存布局接口之一。

为什么现在就做：

- 现在你还在实模式，BIOS 中断还能直接用
- 进入保护模式后，再直接用 BIOS 服务会麻烦很多

所以这一步要放在切模式之前做。

---

## 4. 为什么再建 GDT

保护模式不是“写一条指令就切进去”。

它至少要先有一套合法的段描述符，也就是 `GDT`。

这里先用最小三项：

- null descriptor
- 32 位 code segment
- 32 位 data segment

为什么这么做：

- 现在目标只是先成功进入 32 位保护模式
- 不是马上做复杂分段系统
- 所以用最小平坦模型最稳

---

## 5. 为什么最后才进入保护模式

因为保护模式之后，最方便的 BIOS 实模式服务就不再能直接随手用了。

所以顺序必须是：

1. 先做所有依赖 BIOS 的事情
2. 再切模式

这也是为什么这轮严格按下面顺序来：

1. `stage2 ok`
2. `a20 ok`
3. `e820 ok`
4. `protected mode ok`

每个里程碑都能单独验证问题位置。

---

## 6. 这轮实现后，启动链推进到了哪里

这一轮如果成功，你得到的不是一个完整内核，而是更强的启动链：

```text
BIOS
-> stage1
-> stage2
-> A20 enabled
-> E820 memory map collected
-> GDT loaded
-> protected mode
```

这一步的意义非常大，因为它说明你已经脱离“只有实模式 boot sector”的阶段了。

---

## 7. 下一步为什么不是马上 long mode

long mode 确实是目标，但现在直接跳过去会把问题堆太多。

在保护模式和 long mode 之间，你最好先补这些：

1. 更稳定的 32 位打印与调试
2. 更清楚的内存映射数据结构
3. 内核加载地址规划
4. 页表准备

原因是：

> long mode 的前置条件比保护模式更多，如果中间层不稳，很容易一黑屏就完全失去定位能力。

所以这一步先把保护模式站稳，是更合理的做法。

---

## 8. 一个这轮后来暴露出来的真实 loader 坑

这份 stage2 文档后来又多了一个很重要的“复盘点”：

> 当 `kernel.bin` 变大到跨柱面读取时，CHS 读盘代码很容易在细节上写错。

这次项目继续往后做，
内核加了更多模块后，
`kernel.bin` 从原来较小的体积长到了 `30` 个扇区。

而 stage2 里的内核加载从第 `10` 个扇区开始读：

- 第 10 到第 18 个扇区：磁头 0 剩余的 `9` 个扇区
- 第 19 到第 36 个扇区：磁头 1 的 `18` 个扇区
- 加起来刚好 `27` 个扇区

也就是说：

> 第 `28` 个内核扇区开始，读盘逻辑必须从“柱面 0”切到“柱面 1”。

后来暴露出的 bug 就在这里：

1. 代码先算出了 `cylinder`
2. 先把它写进了 `CH`
3. 但后面为了再算 `head/sector`，又执行了 `mov cx, 18`
4. 这条指令会把整个 `CX` 都覆盖掉
5. 于是 `CH` 里刚保存的 `cylinder` 也被一起清掉了

结果就是：

- 前 `27` 个扇区还能正常读
- 一到第 `28` 个扇区，就没有真的进到下一柱面
- 后面的数据开始读错

这个 bug 最迷惑人的地方在于：

- stage2 仍然可能打印 `kernel loaded ok`
- 内核也仍然能“部分启动”
- 但后面某些字符串、表项、只读数据会突然乱码

所以这一轮后来专门把代码改成：

1. 先把 `cylinder` 暂存在 `BX`
2. 等 `head/sector` 算完
3. 最后再把 `BL` 写回 `CH`

为什么这里值得专门记住？

因为它是一个非常典型的 bootloader 调试教训：

> 早期启动代码里，很多 bug 不是“完全起不来”，而是“只在规模变大一点后才暴露出来”。

---

## 9. 后来又踩到一个更隐蔽的 64 KiB 装载边界

再往后继续加模块时，
`kernel.bin` 不只会跨柱面，
还可能继续长到超过：

- `64 KiB`

这时 stage2 里另一个很容易忽略的问题就会暴露出来：

```text
BIOS int 13h 的目标地址是 ES:BX
```

也就是说，
我们不是直接拿一个 32 位线性地址递增，
而是在拿：

```text
segment:offset
```

这套实模式地址表示法往后推数据。

假设内核从：

```text
0x1000:0x0000
```

开始装，
每读 1 个扇区就把偏移 `+512`。

当前 128 个扇区刚好是：

```text
128 * 512 = 65536 = 64 KiB
```

这意味着：

- 如果刚好只读 128 个扇区
  最后一次写完以后 offset 再回卷，其实还不会伤到已经写好的内容
- 但只要再多 1 个扇区
  第 129 个扇区就会重新写到偏移 `0x0000`

如果这时 segment 没跟着前进，
结果就是：

```text
第 129 个扇区覆盖第 1 个扇区
```

表面现象通常会是：

- `long mode ok` 已经打印出来了
- 但一跳到内核入口就坏掉
- 因为内核开头那几百字节已经被后面的扇区覆盖了

所以后来 stage2 的读盘推进逻辑又补了一步：

1. 每轮先把 `offset + 512`
2. 如果 16 位 offset 发生回卷
3. 就把 `segment` 再加 `0x1000`

为什么是 `0x1000`？

因为实模式线性地址公式是：

```text
linear = segment * 16 + offset
```

而 `offset` 一次跨过去的是：

```text
0x10000
```

要让线性地址继续连续，
segment 就必须补上：

```text
0x10000 / 16 = 0x1000
```

这也是一个非常值得记住的 bootloader 教训：

> 你以为你在“往后写 512 字节”，
> 其实你是在同时维护一套会回卷的 `segment:offset` 地址对。

---

## 10. 这一步后来继续推进成了什么

后面的实现已经继续往下做了，不再只停在保护模式。

现在后续已经补上了：

1. 最小页表
2. `PAE`
3. `EFER.LME`
4. 打开分页
5. 跳进 64 位代码段

也就是说，启动链后来已经从：

`... -> protected mode`

继续推进成：

`... -> protected mode -> paging -> long mode`
