# 从命令历史到行编辑 + 历史浏览 + 内核观察命令

这份文档讲的是：

- 为什么 `history` 做完以后，下一步最合理的是补“像命令行一样的编辑能力”
- 为什么方向键 / Home / End / Delete 不能继续假装成普通字符
- 为什么 `history` 浏览不是 shell 单独做，而是 console 和 shell 协作
- 为什么这一轮新增的观察型命令选的是 `bootinfo` / `e820` / `cpu`
- 为什么 shell 提示符和正文开始用不同颜色

---

## 1. 为什么这一步是自然的下一步

做到上一轮时，
你的 shell 已经会：

- 读一整行
- 执行命令
- 记录最近历史

但它还有两个很明显的缺口：

### 缺口 1：历史只能“看”，不能“拿回来用”

你已经能执行：

```text
history
```

看见自己敲过什么，
但如果你想重新执行上一条命令，
还是得手工再打一遍。

这说明现在的历史还只是“静态记录”，
不是“真正参与交互”。

### 缺口 2：当前输入行还不是真正的可编辑命令行

之前的控制台输入本质上只有两种编辑：

- 末尾追加字符
- 末尾退格删除

这就意味着它更像：

> “会回显的文本输入框”

而不像真正命令行。

所以这一轮最自然的下一步就是把它推进成：

> 可以在行中间移动、删除、回看历史、重新编辑再执行的最小命令行。

---

## 2. 为什么方向键不能继续塞进“字符缓冲区”

如果你只从表面看，
会觉得方向键也不过是一次键盘输入，
是不是还能沿用原来的：

```text
扫描码 -> 字符 -> 字符缓冲区
```

答案是不行。

因为方向键本质上不是“要插入到命令行里的可见字符”，
它是：

- 移动光标
- 浏览历史
- 删除当前位置字符

这类行为键。

所以这一轮做的第一个关键升级不是 shell，
而是键盘层：

> 键盘不再只产出字符，而是开始产出“输入事件”。

你可以把它理解成：

```text
以前：
键盘说“我给你一个字符”

现在：
键盘说“我给你一个输入动作”
```

动作可能是：

- 普通字符
- `ArrowUp`
- `ArrowDown`
- `ArrowLeft`
- `ArrowRight`
- `Delete`
- `Home`
- `End`

这就是为什么这轮会新增 `KeyboardInputEvent`。

---

## 3. 为什么要特别处理 `0xE0` 扩展扫描码

你前面做的普通字母数字键，
大多数都可以直接按 Set 1 的单字节扫描码翻译。

但方向键这类特殊键，
经常不是单独一个“普通扫描码”，
而是这种模式：

```text
0xE0 + 后续字节
```

这里的 `0xE0` 可以理解成：

> “注意，接下来这个不是普通字符键，而是扩展键。”

所以键盘驱动这一轮多了一个非常典型的小状态：

- 是否刚刚收到过 `0xE0`

如果刚收到，
那下一个字节就要走“扩展键翻译表”，
而不能再走原来的字母数字字符表。

这一步很重要，
因为它让你第一次真正看到：

> 键盘输入不总是“一个字节直接等于一个字符”，而是可能需要一个很小的状态机。

---

## 4. 为什么行编辑主要放在 console，而不是 shell

很多人第一次做这种功能时，
会很想把：

- 左右移动
- 删除
- Home / End
- 历史浏览

都塞进 shell。

但这里更合理的分层是：

### `keyboard/`

负责：

- 把扫描码翻译成输入事件

### `console/`

负责：

- 当前输入行长什么样
- 光标现在在哪
- 插入 / 删除后屏幕怎么重画

### `shell/`

负责：

- 当前有哪些历史命令可回看
- 当前这一行最终要执行什么命令

所以这一轮真正的职责分配是：

> shell 提供“历史内容”，console 负责“把历史拿来显示和编辑”。

这就是为什么 `console_read_line_with_history()` 会比老的 `console_read_line()` 多一个 history provider。

---

## 5. 为什么这里用“整行重画”而不是做复杂屏幕局部更新

实现行编辑时，
你有两条路：

### 路线 A：每次只更新最小变化区域

优点：

- 看起来更像成熟终端

缺点：

- 实现复杂
- 很容易在早期阶段把光标逻辑写乱

### 路线 B：每次编辑后把这一整行重新画一遍

优点：

- 简单
- 容易验证
- 很适合教学阶段

这轮选的是路线 B。

原因不是“不会做更细粒度更新”，
而是：

> 现在最重要的是先把“缓冲区内容、光标位置、屏幕表现”三者的对应关系写对。

当前项目里，
读行缓冲本来就很短：

- shell 行缓冲 32 字节
- console 行缓冲 32 字节

所以这一轮直接整行重画，成本很低，但逻辑非常清楚。

## 6. 上下方向键浏览历史到底是怎么接起来的

这一轮的浏览逻辑可以概括成：

```text
用户按 Up
-> console 发现这是历史浏览动作
-> console 向 shell 要“第 N 条历史文本”
-> console 把这一整行画出来
-> 用户还可以继续改
```

这里最关键的一点是：

> console 不自己保存命令历史，它只是临时拿 shell 的历史来显示。

这让分层保持得很干净：

- 历史数据归 shell
- 行内光标和重绘归 console

另外这一轮还专门保存了“当前草稿”。

也就是说，
如果你本来已经打了一半命令，
然后按 `Up` 去看老命令，
再按 `Down` 回来，
原来的草稿还能回来。

这一步虽然看起来小，
但它已经很像真正命令行的行为了。

---

## 7. 这一轮的行编辑具体支持什么

现在输入一行命令时，
你已经可以用：

- `Left`
- `Right`
- `Home`
- `End`
- `Backspace`
- `Delete`
- `Up`
- `Down`

来编辑当前行。

你可以把它理解成：

### 左右键

只移动光标，
不改内容。

### `Home / End`

直接跳到这一行的开头/结尾。

### `Backspace`

删除光标左边那个字符。

### `Delete`

删除光标当前位置那个字符。

### `Up / Down`

在当前 shell 历史里往旧/往新浏览。

这就说明你的 shell 现在已经不再是“只会末尾输入”的 demo，
而是开始具备真正命令行的骨架了。

## 8. 为什么这轮新增的观察命令选 `bootinfo` / `e820` / `cpu`

这一轮不是继续乱加很多命令，
而是只补最有学习价值、最贴近当前启动链的观察命令。

### 1. `bootinfo`

它回答的是：

> “stage2 交给 64 位内核的那份最小启动信息，现在到底长什么样？”

这能让你直接看到：

- `magic`
- E820 条数
- 每条 E820 的大小
- E820 数组地址

也就是说，
它第一次把“bootloader 和 kernel 的交接面”直接暴露成可查询状态。

### 2. `e820`

它回答的是：

> “BIOS 给的内存地图，现在在内核里到底是什么样？”

虽然启动日志里早就会打印一次，
但 shell 命令的意义不一样：

> 现在你可以在内核已经跑起来以后，随时再问它一次。

这让 E820 从“只在启动时闪过去的一段日志”，
变成了“运行时可查询的系统状态”。

### 3. `cpu`

它回答的是：

> “当前 CPU/虚拟 CPU 到底向我报告了什么最基本的身份信息？”

这里先只查最适合教学的最小集合：

- vendor string
- 最大 basic leaf
- 最大 extended leaf
- long mode 支持位

这样你第一次把：

- `cpuid`
- long mode 能力位

从“启动时默认假定存在”变成了“运行时能主动查询出来”的信息。

---

## 9. 为什么这轮还顺手把终端配色调轻了

这一步看起来像“小美化”，
但它其实也和调试体验直接相关。

之前终端区域基本是整片亮白字，
在 QEMU 的 VGA 文本界面里会显得有点重，
长时间看很容易觉得刺眼。

所以这一轮顺手做了一个很小但很实用的调整：

- 正文改成更柔和的浅灰色
- 提示符 `os64>` 单独改成轻一点的青色

这样做的目标不是花哨，
而是：

> 让“哪里是正文、哪里是交互入口”更清楚，同时减轻整屏高亮白字带来的压迫感。

这也是为什么 `shell` 这一轮没有直接去写 VGA 颜色细节，
而是继续通过输出回调把“提示符颜色”和“正文颜色”传给 console。

## 10. 这一轮自动测试为什么很有价值

这轮 smoke test 不只是多加几个命令断言，
而是故意把新的交互能力也一起测进去了。

最有代表性的两段是：

### 测试 1：编辑一条被故意打错的命令

测试里不是直接注入：

```text
echo hi42
```

而是先注入一个故意带多余字符的版本，
再通过：

- `Home`
- `Left / Right`
- `Delete`
- `End`
- `Backspace`

把它真正修成：

```text
echo hi42
```

这说明测试验证的不是“echo 命令存在”，
而是：

> 行编辑能力本身真的生效了。

### 测试 2：用上下方向键把 `history` 从历史里再调出来

这里测试的不是“history 命令能打印”，
而是：

> 浏览历史 -> 回显到当前输入行 -> 再执行

这条链真的打通了。

所以这轮测试已经开始像真正命令行回归测试了，
而不是只测几个静态命令字符串。

## 11. 这一轮完成后，你真正学到了什么

这轮表面上看是 shell 变得更好用了，
但本质上你其实在学 5 件很核心的内核交互层思维：

1. 输入事件化
   键盘不一定产出字符，也可能产出动作。
2. 小状态机
   `0xE0` 扩展扫描码要求驱动保留一点点上下文。
3. 分层
   历史内容归 shell，编辑和重绘归 console。
4. 编辑模型
   命令行不仅有“内容”，还有“光标”。
5. 可观察性
   `bootinfo/e820/cpu` 把启动链和机器信息变成运行时可查询数据。

---

## 12. 下一步最合理做什么

做到这里以后，
下一步已经不太适合继续只堆编辑键了。

更合理的方向通常有两个：

1. 再补少量控制型命令
   比如 `reboot`、`panic`
2. 把 shell 继续变成真正的“内核监控面板”
   比如分页状态、堆块统计、页表摘要

如果按“最适合继续学懂”的顺序，
我更建议先做：

> `reboot` 和 `panic`。

因为这会让 shell 从“观察器”再往前迈一步，
第一次变成真正能触发内核控制动作的入口。

一句话总结这一轮：

> 现在这个 shell 已经不只是能执行命令、保存历史了，它开始具备真正命令行的编辑手感，同时还能直接观察 bootloader 到内核这条启动链里的关键状态。
