# 从带参数命令到命令历史

这份文档讲的是：

- 为什么 `echo` / `uptime` 之后，下一步最自然的是 `history`
- 为什么命令历史要放进 `ShellState`
- 为什么这里用固定大小 ring buffer，而不是先上动态分配
- 这一轮测试到底在验证什么

---

## 1. 为什么下一步先做 `history`

现在你的 shell 已经会：

- 执行固定内建命令
- 查询堆、中断、时间这些内核状态
- 解析 `echo <text>` 这种带参数命令

但它还有一个明显特点：

> 每次执行完一条命令，它就把“刚刚发生了什么”忘掉了。

这在最小 shell 阶段完全没问题，
可是一旦你开始连续调试内核，
你很快就会想知道：

- 我刚才到底敲过什么？
- 是不是前面已经执行过 `mem` 或 `irq`？
- 这次问题是在第几条命令之后出现的？

所以 `history` 的意义不是“让 shell 看起来更像 Linux”，
而是：

> 让 shell 第一次开始管理自己的状态。

这一步非常重要，
因为它意味着 shell 不再只是“收到一行就执行”，
而是开始记住和组织自己的交互历史。

---

## 2. 为什么历史记录要放进 `ShellState`

这一轮最重要的设计决定是：

> 不额外塞一个全局 `g_history`，而是把命令历史直接放进 `ShellState`。

为什么这样做更合理？

因为历史记录本质上就是 shell 自己的内部状态。

它不是：

- 页分配器状态
- 定时器状态
- 键盘驱动状态

它是：

- “这个 shell 最近收到了哪些命令”

所以最自然的归属位置就是：

- `ShellState.history_*`

这样有三个直接好处：

### 1. 模块边界更清楚

看结构体你就知道：

```text
ShellState
-> 输出回调
-> allocator / heap 引用
-> history ring buffer
```

也就是说：

> shell 需要什么状态，就在自己的状态对象里显式写出来。

### 2. 测试更自然

因为 smoke test 本来就已经在初始化同一个 `g_shell`，
现在历史记录自动跟着这份状态走，
不用再额外拼旁路。

### 3. 后面更容易扩

以后如果你还想加：

- 历史条数上限
- 命令别名
- 当前工作上下文
- 简单环境变量

这些都可以继续往 `ShellState` 身上挂，
不会散落成很多难找的全局变量。

---

## 3. 为什么这里先用固定大小 ring buffer

你可能会问：

> “既然已经有堆了，为什么不直接给 history 动态分配？”

原因不是“不能”，
而是“现在没必要”。

这一轮更重要的是先把历史功能的主链和数据结构想清楚，
而不是一上来就把它做成复杂容器。

所以这里先用了一个很典型的内核早期方案：

> 固定大小数组 + ring buffer（环形缓冲区）

它的意思是：

- 最多只记最近 N 条命令
- 新命令来了就写入“下一格”
- 如果写满了，就从头覆盖最旧那条

这套方案现在特别适合你这个项目，
因为它有几个优点：

### 1. 完全确定

内存布局固定，
不会因为 history 本身再去触发复杂堆分配行为。

### 2. 很容易验证

你只要看：

- `history_count`
- `history_next_slot`
- 输出顺序

就能判断 ring buffer 对不对。

### 3. 非常像真实内核早期写法

在很多内核早期模块里，
第一版都会先选这种：

- 固定容量
- 无碎片
- 行为可预测

而不是一开始就上“无限扩容”的复杂结构。

---

## 4. 为什么 `history` 要记录当前这条 `history` 命令自己

这一轮有一个容易被忽略、但很值得讲清楚的小点：

> 当你执行 `history` 时，这条 `history` 命令本身也会被记进历史里。

为什么这样做？

因为从“命令历史”的角度看，
`history` 本身也是用户真实执行过的一条命令。

所以执行顺序变成：

```text
1. 先把当前非空命令写进 history
2. 再解析它是什么命令
3. 如果它正好是 history，就把当前完整历史打印出来
```

这样输出时你就会看到：

- 前面执行过的命令
- 以及当前这条 `history`

这对小白也更容易理解，
因为它符合直觉：

> “我刚刚确实敲了 history，那它就应该出现在历史里。”

---

## 5. 这一轮具体新增了什么字段

现在 `ShellState` 里除了原来的：

- `allocator`
- `heap`
- `output`

还多了这几类字段：

- `history_count`
  当前缓冲区里真的有多少条命令
- `history_next_slot`
  下一条命令应该写到哪里
- `history_total_count`
  shell 从初始化以来一共收过多少条非空命令
- `history_sequence_numbers[]`
  每个槽位当前保存的是第几条命令
- `history_entries[][]`
  每条历史命令自己的文本

你可以把它理解成：

```text
history_entries
是“内容”

history_sequence_numbers
是“编号”

history_count / history_next_slot / history_total_count
是“这个 ring buffer 现在处于什么状态”
```

---

## 6. 为什么命令文本要先 trim 再存

这一轮在写入 history 之前，
会先做两件事：

1. 去掉前导空格
2. 去掉尾部空格

为什么这样做？

因为 shell 真正执行命令时，
本来也会把前导空格跳过去再识别命令名。

如果 history 完全原样保存，
你可能会得到这种东西：

```text
history[6]=   echo hi42
```

这虽然“忠实记录了原输入”，
但对现在这版教学内核并不友好。

所以这里先选择一个更利于理解的策略：

> history 记“规范化后的命令行”，而不是保留所有外层空白。

这样打印出来更整齐，
也更适合你现在拿它做调试面板。

---

## 7. `history` 命令到底输出什么

这一轮 `history` 会先打印两类总览信息：

- `history_buffered=...`
- `history_total=...`

它们分别表示：

### `history_buffered`

当前 ring buffer 里还实际保留着多少条命令。

因为缓冲区有上限，
所以这个数字最多只到固定容量。

### `history_total`

从这次 shell 初始化以来，
一共执行过多少条非空命令。

这个数字不会因为覆盖旧命令而回退。

然后再按“从最旧到最新”的顺序打印：

```text
history[1]=help
history[2]=mem
...
```

这里的方括号里不是“当前数组下标”，
而是：

> 这条命令在整个 shell 生命周期里是第几条命令。

这就是为什么即使以后 ring buffer 覆盖了旧命令，
你仍然能看出“它原本是第几次输入”。

---

## 8. 这一轮自动测试为什么很有价值

这轮 smoke test 不是只验证：

- `history` 这个单词能被识别

而是继续验证更完整的链路：

```text
前面先执行 help/mem/ticks/heap/irq/echo/uptime
-> history 把这些已执行命令都记下来
-> history 再把它们按顺序打印出来
```

所以它实际上一次性验证了 3 件事：

### 1. 记录时机对不对

`history` 打印时，前面命令都已经进历史里了。

### 2. 顺序对不对

输出不是乱序，
而是按从旧到新的用户视角打印。

### 3. 当前命令是否也被记录

测试里会直接断言：

```text
history[8]=history
```

这证明当前这条 `history` 自己也进入了历史。

---

## 9. 这一轮完成后，你真正学到了什么

这一轮表面上只是多了一个 `history` 命令，
但本质上你其实在学 4 件更底层的事：

1. 状态归属
   某个功能的状态应该挂在哪个模块身上。
2. 固定容量数据结构
   不是所有功能都要先上动态内存，早期内核很多地方先用 ring buffer 更稳。
3. 命令生命周期
   一条命令不是“解析一下就完了”，它还可能需要被记录、回放、统计。
4. 自动回归
   新功能不是手工试一下，而是继续放回完整键盘输入链路里验证。

---

## 10. 下一步最合理做什么

有了 `history` 之后，
下一步最自然的通常是把“看见历史”推进到“重新利用历史”。

最合理的方向有两个：

1. 上下方向键浏览历史
2. 更好的行编辑
   比如左右移动光标、删除键、插入位置编辑

如果按“最适合继续学懂”的顺序，
我更建议先做：

> 上下方向键浏览历史。

因为这样你会第一次把：

- 键盘扫描码
- 控制台行缓冲
- shell 历史状态

真正连成一个更像命令行的交互系统。

一句话总结这一轮：

> 现在这个 shell 不只会执行命令了，它开始记住自己做过什么，这就是命令行从“输入器”向“交互系统”迈出的下一步。
