# 从最小字符输入到控制台行输入

这份文档讲的是这一轮为什么继续做：

- VGA 控制台输出模块
- 控制台字符回显
- 最小 `console_read_line()`
- 退格修正
- 自动化整行输入测试

---

## 1. 为什么“能读字符”还不够

上一轮你已经有了：

- 键盘 IRQ
- 扫描码到字符的最小翻译
- 字符环形缓冲区
- `keyboard_try_read_char()`

也就是说，
内核已经能回答：

> “下一个字符是什么？”

但它还不能回答另一个更接近“真实终端”的问题：

> “用户这一整行到底输入了什么？”

更具体地说，
现在还缺两层能力：

1. 字符应该回显到屏幕上
2. 回车之前的一串字符，应该被拼成一整行

所以这一轮的目标就是把流程从：

```text
按键
-> 字符
-> 字符缓冲区
```

继续推进成：

```text
按键
-> 字符
-> 字符缓冲区
-> VGA 回显
-> 一整行输入
```

---

## 2. 为什么单独拆一个 `console/` 模块

如果继续把这些逻辑都塞进 `kernel_main.cpp`，
很快你就会分不清：

- 哪些代码是在做测试
- 哪些代码是在做真正的控制台能力

所以这一轮专门新增了：

- `kernel/console/console.hpp`
- `kernel/console/console.cpp`

这样后面再往上做：

- `getc()`
- `readline()`
- 最小 shell

时，结构就更清楚。

一句话理解：

> `interrupts/keyboard` 管“输入从硬件进来”，`console/` 管“输入怎么显示、怎么读成一整行”。

---

## 3. 这一轮控制台模块具体做了什么

### 1. 维护自己的光标位置

现在控制台自己保存：

- 当前行
- 当前列
- 起始行
- 颜色

为什么还要有“起始行”？

因为前面几行已经被用来显示：

- `idt ok`
- `page allocator ok`
- `keyboard ok`

如果控制台也从第 0 行开始乱写，
这些状态信息就会被覆盖。

所以这轮控制台是从更下面的区域开始工作，
只把下面那几行当成自己的输入/输出区域。

### 2. 支持最小字符输出

现在 `console_write_char()` 先支持 3 类字符：

- 普通可打印字符
- `\n` 换行
- `\b` 退格

这已经足够支撑：

- 输入回显
- 行编辑里的退格删除
- 回车结束一行

### 3. 支持简单滚屏

如果控制台一直往下写到屏幕底部，
它会把自己的显示区域整体上卷一行。

为什么这样做？

因为你不希望：

- 光标越界写到屏幕外

也不希望：

- 把前面固定状态行一起卷没了

所以这一轮的滚动范围只限于控制台自己的区域。

---

## 4. `console_read_line()` 到底做了什么

这是这一轮最关键的接口：

```cpp
size_t console_read_line(char* buffer, size_t capacity);
```

它的行为可以按小白思路理解成：

1. 先不停从键盘字符缓冲区里拿字符
2. 普通字符就：
   - 放进当前行缓冲区
   - 同时回显到 VGA
3. 如果收到退格：
   - 删掉当前行最后一个字符
   - 屏幕上也退格
4. 如果收到回车：
   - 屏幕换行
   - 给字符串补 `\0`
   - 返回整行长度

这就意味着：

> 现在内核第一次真正拥有了“读一整行输入”的能力。

---

## 5. 为什么这里要做“回显”

如果没有回显，
用户按了键之后屏幕完全没变化，
那就算内核内部实际上已经收到字符，
从交互体验上看也像“系统没反应”。

所以控制台系统最基础的一件事就是：

> 用户按什么，就先在屏幕上看到什么。

这一轮回显虽然还是最小版，
但它已经具备了交互系统最核心的第一感觉：

- 输入不是黑盒了
- 用户能看见自己刚才输入了什么

---

## 6. 为什么退格处理特别重要

如果只会追加字符，
那用户一旦打错一个字，
这一行就没法修正。

所以这一轮专门补了：

- `console_write_char('\b')`

它做的事情很直接：

1. 光标往前退一格
2. 用空格把那个位置擦掉

注意这里处理的是：

> 显示层面的退格

而 `console_read_line()` 里还会同时处理：

> 行缓冲区层面的退格

也就是说，
退格这件事现在已经在两层同时生效：

- 屏幕上删掉
- 内存里的当前输入行也删掉

---

## 7. 这一轮自测为什么选 `os 65\b4\n`

测试里注入的不是一串随便的字符，
而是专门设计过的一组输入：

```text
o
s
space
6
5
backspace
4
enter
```

最终期望结果是：

```text
os 64
```

为什么这组测试值很合适？

因为它一次就能验证：

1. 字母输入
2. 空格输入
3. 数字输入
4. 退格是否真的改掉了前一个字符
5. 回车是否真的结束了一整行

如果退格没生效，
最后结果就会变成：

- `os 654`

所以这个例子很容易看出对不对。

---

## 8. 这一轮日志重点看什么

这一轮通过后，
串口日志里应该会新增：

- `console_line_length=5`
- `console_line=os 64`
- `console_buffer_remaining=0`
- `console input ok`

它们分别表示：

1. 最终读到的有效字符长度是 5
2. 这一整行内容确实是 `os 64`
3. 行读取结束后，键盘字符缓冲区已经消费干净
4. 整个控制台输入链路测试通过

---

## 9. 这一轮完成后，你真正多了什么能力

现在内核已经不只是：

- 会收到键盘 IRQ
- 会读出字符

而是继续多了：

1. 会把输入回显到 VGA
2. 会处理最小退格编辑
3. 会把一串按键拼成真正的一整行

这一步很关键，
因为后面很多更高层功能都依赖它：

- 最小终端
- 命令解析
- shell
- 调试命令入口

---

## 10. 这一版还缺什么

现在它仍然只是“最小控制台行输入”版本，
所以还有这些限制：

1. 还没有命令解析
2. 还没有历史记录
3. 还没有左右移动光标
4. 还没有 Tab 补全
5. 还没有真正的 shell 循环

这都正常，
因为这一轮目标只是：

> 先把“字符输入”升级成“整行输入”。

---

## 11. 怎么测试这一轮

这一轮继续跑这 3 个命令：

```bash
make test-stage1
make test-invalid-opcode
make test-page-fault
```

为什么后两个也还要跑？

因为这轮虽然主要新增的是控制台模块，
但它也改到了：

- 内核主初始化流程
- 键盘输入的上层消费逻辑
- 构建脚本

所以还是要确认：

> 整行输入做出来之后，异常链和之前的基础设施没有回退。

---

## 12. 下一步最合理做什么

有了最小整行输入之后，
下一步最自然的通常就是：

1. 做最小命令解释器
2. 支持一两个内建命令
3. 把系统信息打印成真正可交互的 shell 输出

如果按“最适合继续学懂”的顺序，
我建议下一步优先做：

> 最小 shell：提示符 + 读一行 + 识别 2~3 个内建命令。

一句话总结这一轮：

> 现在内核不只是能读单个字符，而是第一次真正能像终端一样读完一整行输入了。
