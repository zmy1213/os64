# 从 E820 到第一版页分配器

这份文档专门讲这一轮为什么要继续做：

- 在 `kernel_main` 里真正解析 E820
- 区分 `usable` 和 `reserved`
- 基于 `usable` 区域做第一版 `alloc_page()`

---

## 1. 这一步到底在补什么

前一轮你已经做到：

- bootloader 把控制权交给了 64 位 C++ 内核
- `kernel_main()` 已经真的跑起来了

但当时内核其实还不会做一件最基础的事：

- 不知道“哪些物理内存能用”

也就是说，虽然 `BootInfo` 已经把 E820 传进来了，
但内核还只是检查：

- 指针在不在
- `magic` 对不对

却没有真的把那份内存地图读出来。

所以这一轮的本质就是：

> 让内核开始真正理解 BIOS 给它的物理内存地图。

---

## 2. 为什么这一步比文件系统和调度器更优先

因为后面的很多东西都建立在一个前提上：

- 你已经知道哪些物理内存可以分配

比如：

- 新页表要用内存
- 内核堆要用内存
- 进程/线程结构体要用内存
- 文件缓存要用内存

如果你连“哪段 RAM 真能用”都没搞清楚，
后面很多模块都会建立在沙地上。

所以优先级正确顺序应该是：

```text
BootInfo
-> 解析 E820
-> 找出 usable 区域
-> 第一版页分配器
-> 再继续做更多内核模块
```

---

## 3. 现在内核到底会打印什么

这一轮完成后，串口里会多出三类很重要的信息：

### 1. 每条 E820 记录

例如：

```text
e820[0] base=0x... length=0x... raw_type=0x... kind=usable
```

它的作用是：

- 让你看到 BIOS 真实返回的内存地图

### 2. 真正进入页分配器的 usable 区间

例如：

```text
allocator_range[0] start=0x... end=0x...
```

它的作用是：

- 让你看到“并不是所有 usable 记录都会原样拿去分配”
- 因为我们还会故意跳过低地址危险区

### 3. `alloc_page()` 的结果

例如：

```text
alloc_page_0=0x...
alloc_page_1=0x...
alloc_page_2=0x...
```

它的作用是：

- 证明第一版物理页分配器已经真的能工作了

---

## 4. 为什么还要区分 usable 和 reserved

因为 BIOS 返回的内存不是“全都能拿来随便写”。

里面常见会混着这些区域：

- 可用 RAM
- BIOS 保留区
- 显存/设备洞
- ACPI 相关区域

所以第一步必须先把它们粗分成两类：

- `usable`
- `reserved`

这一轮先用最简单规则：

- `type = 1` 就当 `usable`
- 其它类型先统一当 `reserved`

这还不是最终最细致的内存分类，
但对第一版页分配器来说已经够了。

---

## 5. 为什么页分配器先只从 1 MiB 以上开始

虽然 E820 可能会告诉你：

- 低地址也有一部分可用 RAM

但这一区域历史包袱很重，经常混着：

- BIOS 数据
- bootloader
- VGA
- 页表
- 各种传统 PC 低地址杂项区

所以第一版刻意保守一点：

- 只从 `0x100000` 也就是 1 MiB 以上挑页

这样做的好处是：

- 更稳
- 更容易理解
- 更不容易一开始就分到奇怪区域

等后面你的内核更成熟，再回过头做更完整的物理内存管理。

---

## 6. 第一版页分配器到底是什么模型

它不是：

- buddy
- slab
- NUMA-aware allocator

它现在只是最小模型：

1. 扫描 E820
2. 找到 `usable` 区域
3. 把地址按 4 KiB 页对齐
4. 每次 `alloc_page()` 就顺序拿下一页

你可以把它想成：

- “一本按页撕下来的空白本子”

第一次拿：

- 第 1 页

第二次拿：

- 第 2 页

先求能用、能看懂、能验证，不追求一开始就高级。

---

## 7. 这一轮完成后你真正学到了什么

这一步做完，你会真正掌握三件事：

1. `BootInfo` 不是摆设，它是 bootloader 和内核之间的交接结构
2. E820 决定了“哪些物理内存能用”
3. 页分配器的本质，就是从“可用物理内存区间”里一页一页拿地址

这时你就从：

- “内核能启动”

走到了：

- “内核开始有最基础的内存管理能力”

---

## 8. 下一步再做什么最顺

这一步做完以后，最顺的下一步通常是：

1. 把页分配器封装得更完整一些
2. 开始为页表动态申请新页
3. 做 `memcpy / memset` 这类最小 runtime
4. 再继续做更正规的虚拟内存管理

也就是说，后面就开始从：

- “看懂物理内存”

进入：

- “真正管理物理内存”
