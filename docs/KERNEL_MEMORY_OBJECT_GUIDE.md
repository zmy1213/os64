# 从可释放堆到对象分配 + 更正式的内核内存子系统

这份文档专门讲这一轮为什么继续做：

- `kmalloc` / `kfree`
- `kcalloc`
- `knew<T>` / `kdelete<T>`
- 更正式的内核内存分层

---

## 1. 为什么这一步比“立刻做文件系统”更自然

你现在的内核已经有：

- E820 内存地图
- 物理页分配器
- 页表映射
- 可 `free` 的内核堆
- 能交互的 shell

这说明“底层内存真的已经能用了”。

但这里还有一个很明显的断层：

> 现在内核虽然能分配一段原始字节，
> 但还没有一套更正式、更稳定的“上层模块怎么拿内存”的入口。

如果你现在直接跳去做文件系统，
你很快就会在很多地方反复写：

- 我要不要直接碰 `KernelHeap*`
- 这块内存按多少对齐
- 这是原始字节，还是对象
- 对象析构时怎么还回去

这会让文件系统代码一开始就混进太多“内存分配细节”。

所以这一轮最合理的事不是马上做磁盘格式，
而是先把这层补出来：

```text
页
-> 堆
-> 正式的内核分配接口
-> C++ 对象分配
```

这样后面无论你做：

- 文件系统节点
- 目录项缓存
- 进程对象
- VFS 结构体

都能先站在一个更稳定的内存接口之上。

---

## 2. 这一轮到底补了哪一层

前面你已经有三层：

### 第一层：`PageAllocator`

它负责：

- 从 E820 的 usable 区域里挑真正可分配的 4 KiB 物理页

你可以把它理解成：

> “物理内存仓库管理员”

### 第二层：`paging`

它负责：

- 建页表
- 把一个虚拟地址映射到某个物理页

你可以把它理解成：

> “虚拟地址和物理页之间的翻译员”

### 第三层：`KernelHeap`

它负责：

- 在一段专门的堆虚拟地址区间里分小块内存
- 支持 `free`
- 支持空闲区复用和合并

你可以把它理解成：

> “把整页内存切成一小块一小块可用区域的人”

这一轮新增的是第四层：

### 第四层：`kmemory`

它负责：

- 把“默认内核堆”注册成一个全局可用入口
- 提供更像正式内核接口的：
  - `kmalloc`
  - `kmalloc_aligned`
  - `kcalloc`
  - `kfree`
- 再往上提供：
  - `knew<T>(...)`
  - `kdelete<T>(...)`

你可以把它理解成：

> “以后别的内核模块真正要用的那扇门”

---

## 3. 为什么不能让所有模块直接用 `heap_alloc()`

因为 `heap_alloc()` 还是比较“底层”的接口。

它要求调用者自己知道：

- 当前拿的是哪个 `KernelHeap`
- 这次要的对齐是多少
- 这是字节块，还是对象

这在教学早期是很好的，
因为你能看清每一层到底干了什么。

但如果继续往后做模块，
就会出现两个问题：

### 问题 1：调用点太散

如果每个模块都自己传 `KernelHeap*`，
以后代码会越来越散，
你很难说清：

> “内核默认到底用的是哪一套分配入口”

### 问题 2：对象构造和析构没有被接住

`heap_alloc()` 只能给你：

- 一段原始内存

但 C++ 对象真正需要的是：

1. 先拿到一段满足对齐要求的内存
2. 再调用构造函数
3. 释放前先调用析构函数
4. 最后把内存还给堆

所以这一轮才会补：

- `knew<T>(...)`
- `kdelete<T>(...)`

这不是“花哨封装”，
而是把：

```text
分字节
-> 变对象
-> 析构
-> 归还
```

这条链真正补完整。

## 4. `kmalloc` / `kcalloc` / `knew` 分别是什么

### `kmalloc(size)`

它的意思是：

> “给我一块大小是 `size` 的内核内存。”

这一轮里，
它本质上还是走到底层堆：

```text
kmalloc
-> kmalloc_aligned
-> heap_alloc
```

只是上层模块不再需要自己管 `KernelHeap*` 了。

### `kmalloc_aligned(size, alignment)`

它的意思是：

> “给我一块按指定对齐要求分配的内核内存。”

这个接口很重要，
因为后面很多内核对象并不只是“能放下就行”，
还会有对齐要求。

比如：

- 某些表项结构
- 缓冲区
- 页大小对齐的大块区域

### `kcalloc(count, size)`

它的意思是：

> “给我 `count * size` 这么大的一整块，并保证起始内容是 0。”

虽然当前 `heap_alloc()` 已经会把给调用者的区域清零，
但 `kcalloc()` 这个名字本身很重要，
因为它明确表达了调用者的意图：

> “我现在要的是一块按元素个数计算、并且全零初始化的区域。”

### `knew<T>(...)`

它的意思是：

> “帮我分配一个 `T` 对象，并调用它的构造函数。”

它做的事情可以拆成：

1. 按 `alignof(T)` 分内存
2. 用 placement new 在这块内存上构造对象

这就是 C++ 对象层和“原始字节块”真正接上的地方。

### `kdelete<T>(ptr)`

它的意思是：

> “先析构这个对象，再把它的内存还回去。”

它做的事情可以拆成：

1. `ptr->~T()`
2. `kfree(ptr)`

这一步很关键，
因为它第一次把：

- 对象生命周期
- 堆内存生命周期

绑成了一套完整流程。

---

## 5. 为什么这轮没有直接上 slab / buddy

因为现在你最缺的还不是“更强算法”，
而是“更清晰的层次”。

这一轮目标是：

1. 先把内存子系统的分层站稳
2. 先让后续模块能有正式入口可用
3. 先让 C++ 对象分配这条链打通

而不是立刻做：

- buddy allocator
- slab allocator
- 对象缓存
- NUMA
- per-CPU cache

那些东西当然是现代内核会做的，
但现在如果直接跳过去，
你反而会更难分清：

- 物理页分配器在管什么
- 堆在管什么
- 对象分配层在管什么

所以这一轮的核心价值是：

> 先把“层”补齐，再谈“高级优化”。

## 6. 这轮代码里还补了什么可观察能力

这一轮除了新接口，还顺手把堆统计信息补得更正式了一点。

现在 `heap` 已经不只是看：

- `heap_used_bytes`
- `heap_mapped_bytes`
- `heap_free_bytes`

还会看：

- `heap_active_allocations`
- `heap_total_allocations`
- `heap_failed_allocations`

这几个数字的意义分别是：

### `heap_active_allocations`

当前还有多少块分配没有归还。

它最适合用来看：

> “我这轮新写的代码有没有明显泄漏。”

### `heap_total_allocations`

从堆初始化开始，到现在一共成功分配过多少次。

它最适合用来看：

> “当前模块到底有多频繁在向堆要内存。”

### `heap_failed_allocations`

一共有多少次分配失败。

它最适合用来看：

> “接口是不是被错误使用了，或者堆是不是真的不够了。”

## 7. 这轮怎么测试

### 正常启动链测试

运行：

```bash
make test-stage1
```

这一轮新的关键日志包括：

- `kernel_memory_page_allocator=0x...`
- `kernel_memory_heap=0x...`
- `kmalloc_block=0x...`
- `kcalloc_block=0x...`
- `kcalloc_zero_words=4`
- `kobject_sum=0x2222222222222222`
- `kobject_aligned_ptr=0x...`
- `kobject_alignment=64`
- `kobject_ctor_count=2`
- `kobject_dtor_count=2`
- `kernel_memory_active_before=...`
- `kernel_memory_active_after=...`
- `kernel memory ok`

这些日志分别证明：

- 默认内存系统已经注册好了
- 原始字节块分配可用
- “按个数清零”可用
- 对象构造真的发生了
- 高对齐对象也能正确落到堆上
- 对象析构真的发生了
- 这一轮对象测试没有留下活动分配泄漏

### 异常测试

运行：

```bash
make test-invalid-opcode
make test-page-fault
```

这两项不是专门测 `kmemory`，
但它们能顺带证明：

- 补了新的内存层以后
- 启动链、trap、shell 这些旧功能没有被你改坏

## 8. 这一轮完成后，你真正多了什么能力

现在这个内核的内存分层已经变成：

```text
E820
-> PageAllocator
-> Paging
-> KernelHeap
-> kmalloc / kcalloc / kfree
-> knew<T> / kdelete<T>
```

这跟前面相比最大的变化不是“更快”，
而是：

> 以后写更高层模块时，你终于可以先把“对象分配”当成一个正式存在的能力。

这对后面做这些东西都非常重要：

- 文件系统对象
- 路径名缓存
- 打开的文件描述结构
- 进程 / 线程控制块
- VFS 层

---

## 9. 下一步最自然做什么

这一步做完以后，
再往前走就比“直接做文件系统代码”更踏实了。

更合理的下一步通常有两个方向：

### 方向 A：开始做最小磁盘 I/O + 文件系统读取

比如先做：

- 读 ATA/磁盘扇区
- 解析一个最小文件系统格式
- 把文件读进内存

这时你已经有更正式的对象分配层，
文件节点、目录项、缓冲区就不需要再直接碰底层堆细节。

### 方向 B：继续把内存子系统做深

比如继续做：

- `new[]` / 数组对象辅助接口
- 小对象缓存
- 页回收
- 更像 slab 的对象缓存层

如果你的目标是“更现代的内核基础设施”，
这条路也很合理。

一句话总结：

> 这一轮不是在“做更多内存技巧”，而是在把内核从“会分字节”推进到“会分对象”。
