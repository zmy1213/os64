# 从通用 Trap 到 PIC + PIT + 定时器中断

这份文档专门讲这一轮为什么继续做：

- PIC 重映射
- PIT 定时器初始化
- 第一版硬件 IRQ 框架
- 第一版系统 tick

---

## 1. 为什么这一步是自然的下一步

上一轮你已经有了：

- 前 32 个 CPU 异常的通用 trap 框架
- 能 `free` 的第一版正式内核堆

但那一轮处理的都还是：

- CPU 自己内部抛出的异常

比如：

- `page fault`
- `invalid opcode`
- `general protection fault`

这和真正“活起来的内核”还差一层。

因为现代内核不能只会处理“CPU 自己出错”。
它还必须会处理：

- 时钟到了
- 键盘来了
- 磁盘完成了
- 网卡收到包了

这些都属于：

> 外部硬件中断，也就是 IRQ。

所以这一轮最合理的第一步不是马上做调度器，
而是先把：

1. 外部中断入口接通
2. 最重要的一路 IRQ0 时钟中断接通
3. 让内核第一次拥有持续增长的 tick

---

## 2. PIC 到底是什么

先说小白版：

> PIC 是“老式中断分发员”。

经典 x86 机器上，
很多外部硬件不会直接自己找 CPU 说话，
而是先把中断请求送给 PIC，
再由 PIC 告诉 CPU：

- 现在有第几路 IRQ 到了

传统 8259A PIC 一共有两片：

- 主片：管理 IRQ0 ~ IRQ7
- 从片：管理 IRQ8 ~ IRQ15

合起来就是 16 路硬件中断。

---

## 3. 为什么要“重映射” PIC

这是这一步最关键的原因之一。

PIC 上电后的默认中断向量并不适合我们现在的 64 位内核环境。

默认情况下：

- 主片 IRQ0 ~ IRQ7 常常落在 `0x08 ~ 0x0F`
- 从片 IRQ8 ~ IRQ15 常常落在 `0x70 ~ 0x77`

问题在于：

- CPU 已经把 `0 ~ 31` 留给异常和陷阱

其中 `8 ~ 15` 这一段正好和主 PIC 默认范围冲突。

例如：

- `8` 可能表示 `double fault`
- 但如果不重映射，IRQ0 也可能跑到这个区域

这样你根本分不清：

- 这是 CPU 异常
- 还是硬件中断

所以这一轮我们把 PIC 改到：

- 主片 IRQ0 ~ IRQ7 -> `32 ~ 39`
- 从片 IRQ8 ~ IRQ15 -> `40 ~ 47`

一句话记住：

> `0 ~ 31` 留给 CPU 异常，`32 ~ 47` 留给老式 PIC 的 16 路硬件 IRQ。

---

## 4. PIT 又是什么

PIT 的全名是：

- Programmable Interval Timer

你可以把它理解成：

> 一个会按固定频率不断“敲钟”的硬件计时器。

最经典的一路是：

- PIT 通道 0 -> IRQ0

也就是说，
只要你把 PIT 设好，
它就会周期性地通过 IRQ0 打断 CPU。

这正是内核最想要的第一类硬件中断，
因为后面很多能力都依赖它：

- 系统时间 tick
- sleep
- 超时
- 调度器时间片

---

## 5. 这一轮具体实现了什么

这一轮一共做了 5 件关键事。

### 1. 在 IDT 里接入了 IRQ 入口

上一轮 `IDT` 里主要放的是：

- 前 32 个 CPU 异常

这一轮继续加上：

- `32 ~ 47` 这 16 个硬件 IRQ 向量

也就是说，
现在 CPU 不只是出异常时有门可走，
硬件发 IRQ 时也有门可走了。

### 2. 新增了能返回的 IRQ 汇编 stub

异常和硬件中断有一个非常重要的区别：

- 异常测试里我们通常打印完就停机
- 但时钟中断必须处理完再返回

所以这一轮在 `interrupt_stubs.asm` 里新增了：

- `irq_stub_table`
- `irq_stub_0 ~ irq_stub_15`

它们会做这些事：

1. 手工补一个 `error_code = 0`
2. 压入当前 IRQ 对应的向量号
3. 保存通用寄存器
4. 调 `kernel_handle_irq()`
5. 恢复寄存器
6. `iretq` 返回到被打断前的代码继续执行

这里最重要的一句话是：

> 这一轮的 IRQ stub 不是“打完日志就停机”，而是“处理后返回继续跑”。

### 3. 初始化 PIC，并先只放开 IRQ0

这一轮不是一下子把所有外设中断都接进来，
那样太容易把问题复杂化。

我们故意只做最小版本：

- 完成 PIC 初始化
- 完成向量重映射
- 主片只放开 IRQ0
- 从片全部先屏蔽

所以现在真正会进来的外部中断只有：

- 定时器 IRQ0

这是一种非常典型的内核开发节奏：

> 先只开最重要的一路，把整条链打通，再慢慢开更多设备。

### 4. 初始化 PIT 到 100Hz

这一轮把 PIT 先设成：

- `100 Hz`

意思就是：

- 每秒约产生 100 次 tick

为什么先用 100Hz：

- 太低了不明显
- 太高了日志和测试都不够舒服
- 100Hz 很适合教学和早期验证

PIT 的输入时钟大约是：

- `1193182 Hz`

代码里会做：

- `divisor = 1193182 / 100`

再把这个分频值写入 PIT。

### 5. 有了第一版全局 tick 计数器

现在内核里维护了：

- `g_timer_ticks`

每次收到 IRQ0，
就执行：

- `++g_timer_ticks`

这就是最小版的“系统时钟心跳”。

虽然它现在很简单，
但后面很多功能都能基于它继续长出来：

- `sleep(n ticks)`
- 超时等待
- 周期任务
- 调度器时间片轮转

---

## 6. 什么叫 tick

小白版可以直接这样理解：

> tick 就是“时钟敲了一下”的计数。

如果现在 PIT 频率是 `100Hz`，
那就表示：

- 1 秒大约有 100 个 tick
- 10 个 tick 大约是 0.1 秒
- 20 个 tick 大约是 0.2 秒

所以日志里看到：

- `timer_tick=10`
- `timer_tick=20`

本质就是在证明：

- 时钟不是只来一次
- 而是在持续稳定地来

---

## 7. 为什么 IRQ 处理完后一定要发 EOI

EOI 的全名是：

- End Of Interrupt

你可以把它理解成：

> 给 PIC 回一句：“这次中断我处理完了，你可以继续送下一次了。”

如果不发 EOI，
PIC 可能会认为：

- 上一次 IRQ 还没结束

结果就是：

- 后面的同类中断不再继续送进来

所以这一步不是可选优化，
而是传统 PIC 路径里的必要动作。

如果 IRQ 来自从片，
还要注意顺序：

1. 先给从片发 EOI
2. 再给主片发 EOI

因为从片本身是挂在主片下面的。

---

## 8. `hlt` 在这一步为什么很好用

这一轮测试里，
内核不是傻傻地一直空转等 tick。

而是调用了：

- `hlt`

意思是：

- 先让 CPU 睡眠
- 等下一次中断到了再醒

这比忙等更好理解，
也更接近真正内核在空闲时的思路。

所以这一步你看到的逻辑其实是：

1. 打开中断
2. 看看 tick 到没到目标值
3. 没到就 `hlt`
4. 时钟 IRQ 到来
5. tick 增加
6. CPU 被唤醒继续检查

这就是第一版最小“中断驱动等待”。

## 9. 这一轮怎么测试

### 测试 1：正常启动链 + 定时器测试

命令：

```bash
make test-stage1
```

如果这一轮成功，
日志里会看到：

- `pic ok`
- `pit ok`
- `timer_tick=10`
- `timer_tick=20`
- `timer ok`

这说明：

1. PIC 初始化成功
2. PIT 初始化成功
3. IRQ0 确实在持续发生
4. IRQ 处理后确实成功返回了

### 测试 2：invalid opcode 继续可用

命令：

```bash
make test-invalid-opcode
```

这条测试的意义是：

- 证明加了 IRQ 之后，原来的异常链没有被破坏

也就是你仍然应该看到：

- `timer_tick=10`
- `timer_tick=20`
- `invalid opcode smoke`
- `exception=invalid opcode`

### 测试 3：page fault 继续可用

命令：

```bash
make test-page-fault
```

这条测试的意义是：

- 证明 IRQ 路径和 page fault 路径能够并存

预期仍然应该看到：

- `timer_tick=10`
- `timer_tick=20`
- `page fault smoke`
- `exception=page fault`
- `fault_address=0x0000000000900000`

### 一个很实际的注意点

这两条异常测试不要并行跑。

原因不是内核逻辑错了，
而是当前测试脚本会共用：

- `build/disk.img`

并行时一个 QEMU 可能锁住镜像，
另一个又在重建镜像，
这样日志会互相污染。

所以这三条命令应该顺序执行。

## 10. 这一轮完成后，你真正多了什么能力

现在这个内核已经不只是：

- 能进入 64 位
- 能解析 E820
- 能分配页
- 能建页表
- 能抓 CPU 异常

而是继续多了：

1. 第一版硬件 IRQ 接入链
2. 第一版 PIC 管理
3. 第一版 PIT 定时器
4. 第一版全局系统 tick
5. 第一版 `hlt + 中断唤醒` 的等待方式

这一步意义非常大，
因为它标志着你的内核第一次开始被：

- 外部硬件事件

真正驱动起来了。

---

## 11. 这一版的局限也要看清楚

这一轮仍然只是教学版最小实现，
所以还有这些明确限制：

1. 只打开了 IRQ0，还没有键盘、磁盘、串口等其它外设 IRQ
2. 仍然使用经典 8259A PIC，还没有进 APIC / IOAPIC
3. `g_timer_ticks` 还只是一个全局计数器，没有更正式的时间子系统
4. 还没有 `sleep()`、定时器队列、超时唤醒
5. 还没有真正的任务切换和调度器

这些都不是问题，
而是当前阶段故意不一次性展开。

---

## 12. 下一步最合理做什么

有了第一版 timer IRQ 之后，
下一步最自然的通常是下面几类里选一个：

1. 基于 tick 做最小 `sleep` / `delay`
2. 接键盘 IRQ，第一次真正响应用户输入
3. 做一个最小任务结构和时钟驱动的调度器骨架
4. 继续从 PIC 过渡到更现代的 APIC 路线

如果按“最容易接着学懂”的顺序，
我建议下一步先做：

> `tick` 驱动的最小等待接口，或者键盘中断。

一句话总结这一轮：

> 现在内核不只是“CPU 出错时能接住”，而是第一次真正能接住“外部硬件周期性打来的时钟中断”了。
