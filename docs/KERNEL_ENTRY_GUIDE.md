# 从 Long Mode 到 C++ 内核

这份文档专门讲这一轮做的事：

- 为什么 `stage2` 不在 `long mode ok` 就结束
- 为什么现在先用固定地址加载 `kernel.bin`
- 为什么还要多一个 `kernel_entry.asm`
- 为什么要传 `BootInfo`

---

## 1. 这一步到底在解决什么

前一轮我们已经证明了一件事：

- CPU 能从 BIOS 一路走到 64 位 long mode

但这还只是：

- “启动链打通了”

还不是：

- “你已经有自己的内核了”

因为前一轮 long mode 成功后，代码还停留在 `stage2` 里。

所以这一轮真正要解决的是：

> 让 bootloader 和真正的 64 位内核分家。

也就是：

```text
BIOS
-> stage1
-> stage2
-> long mode
-> kernel_entry
-> kernel_main.cpp
```

---

## 2. 为什么现在不急着上 ELF 解析器

严格来说，更“完整”的路线是：

- `stage2` 读入 `kernel.elf`
- 自己解析 ELF program headers
- 再把每个段搬到正确位置

但你现在还是小白阶段，这一步如果直接上，复杂度会一下子抬很高。

你会同时面对：

- 磁盘读取
- ELF 格式
- 段复制
- BSS 清零
- 入口地址解析
- long mode 跳转

这样一旦黑屏，你很难判断到底是哪一层错了。

所以这一轮故意先走“最小闭环”：

- 先把内核链接到固定地址 `0x10000`
- 再把它导出成 `kernel.bin`
- `stage2` 直接把这块原始二进制读到那个地址

这样先把：

- “bootloader 能不能跳到 C++ 内核”

这个核心问题单独解决掉。

---

## 3. 为什么把内核先放到固定低地址

这一轮我把它先放到：

- `0x10000`

也就是 64 KiB。

这么选不是因为它“最现代”，而是因为它当前最省事：

- 高于 `stage1/stage2`
- 还在传统 BIOS 最容易处理的低地址区
- 仍然落在我们当前“恒等映射低 2 MiB”的范围内
- 可以继续用最基础的 BIOS `int 13h` CHS 读盘

这样做的好处是：

- `stage2` 读到哪里，long mode 后还是那个地址
- 你暂时不用再处理更复杂的高地址映射

所以它不是“最终最优地址”，而是：

- 当前最适合学习和调试的地址

后面等你把：

- `bootloader -> kernel`

这条链完全站稳之后，再把内核挪高到 `1 MiB` 甚至更规范的布局，会更合适。

---

## 4. 为什么还在实模式里把内核读进来

因为我们现在读盘依赖的是 BIOS `int 13h`。

而 BIOS 中断服务最稳妥的使用环境就是：

- 16 位实模式

一旦你已经进入保护模式甚至 long mode：

- 你就不能再像前面那样直接用 BIOS 中断读盘了

所以顺序必须是：

1. 还在实模式时把 `kernel.bin` 从磁盘读到内存
2. 再切保护模式
3. 再切 long mode
4. 最后跳进内核

---

这一轮还顺便做了一个取舍：

- 不急着上 BIOS 扩展读盘
- 也不急着上 LBA + DAP

因为当前我们还是想把重点放在：

- 理解启动链本身

而不是一口气又引入：

- BIOS 扩展读盘
- DAP
- ELF 搬运

所以这里仍然继续用最基础的 CHS 读盘。

不过这里要注意一件非常具体的事：

- 现在这张 `disk.img` 是按 `1.44MB` 软盘布局做的
- `stage1/stage2` 里的 CHS 参数也是按软盘几何在算

所以你在 `QEMU` 里启动它时，也要把它当成软盘挂上：

```bash
qemu-system-x86_64 -drive format=raw,file=disk.img,if=floppy,index=0
```

如果你把它当成普通硬盘挂载，前 18 个线性扇区有时还会“碰巧正常”，
但一旦跨到后面，软盘几何和硬盘几何开始不同，loader 就会把后续扇区读错地方。

现在这套 CHS 代码虽然还是最基础的 BIOS 读盘，
但它已经能在软盘几何下继续跨过当前磁道，切到下一个磁头、下一个柱面。

这样你少背一层新接口，更容易先把主线理解透。

---

## 5. 为什么不是直接跳 C++，还要加 `kernel_entry.asm`

因为 C++ 编译器默认假设很多前提已经成立，比如：

- 栈已经可用
- 调用约定已经满足
- 入口不是 BIOS 风格

所以我们先用一个很薄的 64 位汇编入口做这些事：

1. 切到内核自己的栈
2. 保持寄存器状态简单可控
3. 调用 `kernel_main`
4. 如果返回，就统一退出或停机

你可以把 `kernel_entry.asm` 理解成：

- `stage2` 和 `kernel_main.cpp` 之间的转接头

---

## 6. 为什么要传 `BootInfo`

因为内核不是凭空开始运行的，它需要知道启动阶段已经收集了什么。

这一轮最重要的信息就是：

- E820 内存映射地址
- E820 条目数
- 每条记录大小

所以我们先定义一个最小 `BootInfo`：

- `magic`
- `memory_map_count`
- `memory_map_entry_size`
- `memory_map_ptr`

这样 `kernel_main.cpp` 一进来就能确认：

- “我拿到的启动信息是正确的”

以后你再扩展：

- framebuffer
- ACPI RSDP
- 内核命令行
- 物理内存上界

也都可以继续往这个结构里加。

---

## 7. 这一轮完成后你真正得到什么

成功标准不再只是：

- `long mode ok`

而是还要继续看到：

- `hello from os64 kernel`
- `boot info ok`

这代表的意义很大：

- 你已经不只是“会切 CPU 模式”
- 而是“已经能让自己的 64 位 C++ 内核真正跑起来”

---

## 8. 下一步最合理做什么

这一步做完后，最该继续的是：

1. 把 `BootInfo` 里的 E820 数据在内核里真正解析出来
2. 做最小 VGA/serial console 封装
3. 做 `memcpy / memset / strlen` 这类最小 runtime
4. 开始第一版物理页分配器

这时你才算真正进入：

- “内核初始化”

而不再只是 bootloader 阶段。
