# 从最小 IDT 到通用 Trap + 可释放堆

这份文档专门讲这一轮为什么继续做：

- 通用 trap 框架
- 32 个 CPU 异常统一入口
- 非 `page fault` 异常测试
- 可 `free` 的第一版正式内核堆

---

## 1. 为什么上一轮还不够

上一轮你已经有了：

- 最小 IDT
- `page fault` 基础处理
- 第一版内核堆

但那一版还有两个明显限制：

### 限制 1：trap 还只是“点状接线”

上一轮只手工接了几个最关键的异常：

- divide error
- invalid opcode
- double fault
- general protection fault
- page fault

这对“先把最危险的坑接住”已经够用了，
但它还不是一套真正通用的 trap 框架。

因为后面你再想补别的异常时，
还得继续手写：

- 一个新 stub
- 一个新 IDT gate
- 一个新名字映射

这会越来越散。

### 限制 2：堆只能分配，不能回收

上一轮的堆本质上还是：

> bump allocator

也就是：

- 一直往前切
- 没有 `free`
- 不会复用旧空间

这在“内核刚起来、想先证明能分配”时很合理，
但如果再往后走，你很快就会遇到一个现实问题：

> 分过一次的内存永远回不来。

所以这一步最自然的升级就是：

1. trap 变成“统一入口 + 统一分发”
2. heap 变成“有空闲区管理 + 能 free + 能复用”

---

## 2. 什么叫“通用 trap 框架”

先说小白版理解：

> trap 框架就是“CPU 一出异常，先走一条统一主路，再按异常号分流”。

这一轮做的事是：

1. 先把前 32 个 CPU 异常向量都放进 IDT
2. 每个向量都对应一个汇编 stub
3. 每个 stub 最终都跳到同一个 `isr_common`
4. `isr_common` 再把信息交给 C++ 的 `kernel_handle_exception()`

所以它和上一轮最大的区别是：

- 上一轮更像“几根手搓飞线”
- 这一轮开始变成“统一总线”

---

## 3. 这轮 trap 框架具体统一了什么

### 统一点 1：异常向量表

现在 `interrupt_stubs.asm` 里不再只导出几个零散入口，
而是导出一整张：

- `isr_stub_table`

你可以把它理解成：

> 一个“第 0 号异常入口在哪、第 1 号异常入口在哪……”的地址表。

这样 `interrupts.cpp` 初始化 IDT 时，
就不再是一项项手写：

- `set_idt_gate(0, ...)`
- `set_idt_gate(6, ...)`
- `set_idt_gate(14, ...)`

而是可以直接循环把前 32 项都接进去。

### 统一点 2：错误码处理

x86 的异常有个麻烦点：

- 有些异常会自动压 `error code`
- 有些不会

如果你不统一这个差异，
后面的 C++ 处理函数就很难用一份固定结构去读栈。

所以这一轮的 stub 做了一个关键动作：

- 没有错误码的异常，手工补一个 `0`
- 有错误码的异常，只额外压入 `vector`

这样进入 `kernel_handle_exception()` 时，
栈布局就被统一成了同一种 `InterruptFrame`。

这就是“框架”的味道：

- 上层逻辑不再关心每个异常的细节差异
- 统一按同一份结构读数据

### 统一点 3：异常名字表

这轮把前 32 个异常的名字表也收进 `interrupts.cpp` 了。

所以现在不只是能打印：

- `vector=0x000000000000000E`

还可以直接打印：

- `exception=page fault`

或者：

- `exception=invalid opcode`

这对小白非常重要，
因为你第一眼能先看到“中文概念对应的那一类错误”，
而不是先被纯数字吓住。

## 4. 什么叫“可释放的正式堆”

这轮的堆和上一轮最大的区别是：

> 不再只记“下一刀从哪里切”，而是开始真正管理“哪些区域空着”。 

现在内核堆内部有两种元数据：

### 第一种：空闲区头

空闲区头放在每一段空闲区域的开头，
它记录：

- 这段空闲区有多大
- 下一段空闲区是谁

所以整块堆空间现在不是“一片模糊内存”，
而是开始被组织成：

- 一条按地址顺序串起来的空闲区链表

### 第二种：已分配块头

当你真正 `heap_alloc()` 成功时，
返回给调用者之前，
内核会在这块内存前面偷偷放一个“已分配块头”。

里面记录：

- 这次分配实际占掉的整段区域从哪开始
- 这整段区域有多大
- 用户真正请求的 payload 有多大

这样后面 `heap_free(ptr)` 时，
内核才能倒推回来：

- 这一块原来到底占了哪一整段空间

然后把它重新插回空闲区链表。

---

## 5. 这轮堆分配器真正多了哪些能力

### 能力 1：支持 `free`

现在你可以把一块已经不用的堆内存还回去。

这意味着：

- 堆不是“一去不回”
- 内核终于开始像真正的分配器，而不只是演示代码

### 能力 2：支持复用

如果你先分一块小内存，再 `free` 掉，
然后申请一个更小的块，
这一轮分配器会优先尝试从已有空闲区里找位置。

也就是说：

- 它会先用旧空间
- 不会一有请求就立刻继续向后扩堆

这一步很关键，
因为这才是“管理内存”，
而不是“只会消耗内存”。

### 能力 3：支持相邻空闲块合并

光有 `free` 还不够。

如果你释放了两块相邻的小块，
但分配器不会把它们合并，
那么后面来一个稍大的请求时，
还是可能分不到连续空间。

所以这一轮又补了一步：

- 相邻空闲块自动合并

这意味着：

- 释放小块后，不只是“多了两块零碎空闲”
- 而是有机会重新变成一大块连续空间

---

## 6. 为什么这一轮还保留页对齐大块测试

虽然堆已经不只是 bump allocator 了，
这一轮仍然保留了：

- 一个 page-aligned 的大块分配测试

原因是我们不想在升级正式堆的同时，
把上一轮已经有的能力退回去。

也就是说，
现在不只是要验证：

- `free` / 复用 / 合并

还要继续验证：

- 跨页大块分配仍然成立
- 页对齐需求仍然成立

这能证明这轮不是“用新功能换掉旧能力”，
而是在原来基础上往前扩。

## 7. 这一轮怎么测试

### 测试 1：正常启动链

命令：

```bash
make test-stage1
```

这条测试现在会同时验证：

- `heap_large_page_aligned=1`
- `heap_reuse=0x...`
- `heap_coalesced=0x...`

它们分别代表：

1. 大块分配仍然能按页对齐
2. 释放后的小块能复用旧地址
3. 两个相邻释放块能合并后重新分配

### 测试 2：invalid opcode 冒烟测试

命令：

```bash
make test-invalid-opcode
```

这条测试会构建一张单独镜像，
在正常初始化后故意执行：

- `ud2`

`ud2` 是 x86 专门留给“故意触发非法指令异常”的指令。

如果通用 trap 框架真的接通了，
你会看到：

- `exception=invalid opcode`
- `vector=0x0000000000000006`
- `error_code=0x0000000000000000`

这说明：

- 不是只有 `page fault` 才能走这条异常链
- 另一种完全不同的异常也能从同一套入口进来

### 测试 3：page fault 冒烟测试

命令：

```bash
make test-page-fault
```

这条测试会继续故意访问未映射地址：

- `0x900000`

预期日志里会看到：

- `exception=page fault`
- `fault_address=0x0000000000900000`
- `fault_rip=0x...`

这说明：

- trap 框架对 page fault 这类带错误码、带 `CR2` 信息的异常也仍然正常工作

---

## 8. 这一轮完成后，你真正多了什么能力

现在这个内核已经不只是：

- 有几个能用的异常入口
- 有一个只能一路向前长的堆

而是开始具备：

1. 一套前 32 个 CPU 异常的统一接入框架
2. 一套可打印异常名字、向量号、错误码的通用异常路径
3. 一个能 `free`、能复用、能合并空闲块的正式堆分配器雏形

这三件事连起来的意义很大，
因为它们直接决定你后面能不能继续做：

- 更复杂的内核对象分配
- 更完整的异常/陷阱处理
- 定时器中断和调度器骨架

---

## 9. 这一步之后最自然做什么

当你已经有：

- 通用 trap 框架
- 可释放的堆

下一步通常最合理的是：

1. 把异常处理继续扩成更正式的通用 trap 打印框架
2. 开始接 PIC/APIC 和时钟中断
3. 让调度器第一次真正“被时钟驱动起来”

一句话总结这一轮：

> 现在内核不只是“出错能看见、要内存能拿到”，而是开始具备“异常统一进、内存还能还回去”的基础能力。
