# 从 timer tick 到最小 wait / sleep 接口

这份文档专门讲这一轮为什么继续做：

- `timer_wait_ticks()`
- `timer_sleep_ms()`
- `hlt + tick` 驱动的最小等待

---

## 1. 为什么上一轮还不够

上一轮你已经有了：

- PIC
- PIT
- IRQ0 定时器中断
- `g_timer_ticks`

也就是说，
内核已经能知道：

> “时钟又响了一下。”

但它还不能很方便地做一件特别基础的事：

> “那我就等 3 个 tick 再继续。”

或者：

> “那我就大概睡 50ms 再继续。”

这就是上一轮和“真正能开始用时间”的差别。

所以这一轮最自然的下一步就是：

1. 先做一个按 tick 等待的最小接口
2. 再做一个按毫秒等待的更好理解接口

---

## 2. 这一轮到底新增了什么

这一轮在 `pit.cpp/.hpp` 里继续加了 4 个核心能力：

- `timer_frequency_hz()`
- `timer_is_ready()`
- `timer_wait_ticks(uint64_t ticks)`
- `timer_sleep_ms(uint64_t milliseconds)`

你可以把它们理解成两层：

### 第一层：状态查询

- `timer_frequency_hz()`
  看 PIT 现在每秒大约打多少次 tick。
- `timer_is_ready()`
  看 PIT 是否已经初始化成功。

### 第二层：真正等待

- `timer_wait_ticks()`
  直接按 tick 数等待。
- `timer_sleep_ms()`
  先把毫秒换成 tick，再复用 `timer_wait_ticks()`。

---

## 3. `timer_wait_ticks()` 是怎么工作的

这个函数的核心思路非常简单：

1. 先记住开始时的 `start_tick`
2. 计算目标是至少等多少个 tick
3. 只要当前 `g_timer_ticks - start_tick` 还不够
4. 就执行一次 `hlt`
5. 下一次中断把 CPU 唤醒后，再继续检查

也就是：

```text
记住起点
-> 还没到目标
-> hlt 睡眠
-> IRQ0 到来
-> tick + 1
-> CPU 醒来继续看
-> 到目标就返回
```

它最重要的意义不是“很高级”，
而是：

> 这是你第一次真正用“中断驱动的时间”来控制内核行为。

---

## 4. 为什么这里用 `hlt` 而不是空转

如果不用 `hlt`，
最笨的方法是这样：

```text
while (tick 还没到) {
  一直空转
}
```

这当然也能等到目标，
但它会一直疯狂占着 CPU。

用了 `hlt` 之后，逻辑就变成：

- 没事先睡下
- 等中断来了再醒

这更像真正内核的空闲等待方式。

所以这一步你可以理解成：

> 从“忙等时间”进化到“被中断唤醒”。

---

## 5. 为什么 `timer_wait_ticks()` 前必须先开中断

这一点非常重要。

`timer_wait_ticks()` 内部会调用：

- `hlt`

如果你在：

- 中断关闭

的情况下执行 `hlt`，
那 CPU 睡下去之后，
定时器 IRQ 就没法把它正常唤醒回来，
你会看起来像“卡死了”。

所以这个接口的前提条件是：

1. PIT 已经初始化
2. PIC 已经放开 IRQ0
3. `sti` 已经执行，也就是中断已经打开

一句话记住：

> `sleep/wait` 不是单独存在的，它依赖前一轮已经打通的整个 IRQ 链。

---

## 6. `timer_sleep_ms()` 又是怎么工作的

很多时候直接说“等 5 个 tick”，
对于人来说不如“等 50ms”直观。

所以这一轮又加了：

- `timer_sleep_ms(uint64_t milliseconds)`

它内部并不神秘，
本质上就是做一次单位换算：

```text
毫秒 -> 需要多少个 tick
```

公式可以理解成：

```text
ticks = milliseconds * frequency_hz / 1000
```

代码里又多做了一步：

- 向上取整

原因是为了避免这种情况：

- 频率比较低
- 你只等一个很小的毫秒数
- 结果算出来是 0 tick

如果真算成 0，
那就等于“完全没等”。

所以这里故意做成：

> 只要你说要等一小段时间，就至少等到 1 个 tick。

---

## 7. 这一轮测试里具体验证了什么

现在 `kernel_main.cpp` 里的定时器测试已经不只是：

- 看见 `timer_tick=10`
- 看见 `timer_tick=20`

而是继续做了两段等待测试：

### 测试 1：按 tick 等待

代码会先记住开始 tick，
再调用：

- `timer_wait_ticks(3)`

回来之后打印：

- `timer_wait_elapsed_ticks=...`

这说明：

- 内核已经能“至少等 3 个 tick 再继续”

### 测试 2：按毫秒等待

代码会再调用：

- `timer_sleep_ms(50)`

在当前 `100Hz` 下，
`50ms` 大约对应：

- `5` 个 tick

回来之后会打印：

- `timer_sleep_ms=50`
- `timer_sleep_elapsed_ticks=...`

这说明：

- 内核已经能把“毫秒”翻译成“tick”，然后真的等过去

---

## 8. 怎么测试这一轮

命令：

```bash
make test-stage1
```

预期日志里会新增这些内容：

- `pit_frequency_hz=100`
- `timer_wait_elapsed_ticks=...`
- `timer_sleep_ms=50`
- `timer_sleep_elapsed_ticks=...`
- `timer ok`

这里的 `timer ok` 不只是说明：

- IRQ0 来了

还说明：

- 等待接口本身的最小语义也成立了

---

## 9. 这一轮完成后，你真正多了什么能力

现在这个内核已经不只是：

- 能收到时钟中断
- 能把 tick 记下来

而是继续多了：

1. 按 tick 等待
2. 按毫秒等待
3. `hlt + 中断唤醒` 的最小 sleep 语义

这一步非常关键，
因为后面很多更像“操作系统行为”的东西都能建立在它上面：

- `sleep()`
- 超时等待
- 轮询 + 超时退出
- 调度器时间片

---

## 10. 这一版的局限也要看清楚

这仍然只是第一版教学实现，
所以还有这些限制：

1. 现在还只是单 CPU、单全局 tick
2. 还没有真正的时钟子系统
3. 还没有定时器队列
4. 还没有“到点唤醒哪个任务”的机制
5. 还没有调度器，只是当前 CPU 自己睡自己醒

所以它现在更准确的定位是：

> 一个最小可用的时间等待基础设施。

---

## 11. 下一步最合理做什么

有了最小 `sleep/wait` 之后，
下一步最自然的通常是下面几条路之一：

1. 接键盘 IRQ，让内核第一次真正响应外部输入
2. 做一个最小超时接口，比如“最多等 N 个 tick”
3. 做最小任务结构，把 tick 和调度器骨架接起来

如果按“最容易继续学懂”的顺序，
我建议下一步优先做：

> 键盘中断，或者基于 tick 的超时等待。

一句话总结这一轮：

> 现在内核不只是“知道时间在流逝”，而是第一次真正能“等一段时间再继续执行”了。
