# 从控制台行输入到最小 Shell

> 这份文档记录的是 shell 的第一版形态：`help / mem / ticks`。
> 当前仓库已经继续扩到 `heap / irq / clear / uptime / echo`，建议接着看 [KERNEL_SHELL_EXPANSION_GUIDE.md](./KERNEL_SHELL_EXPANSION_GUIDE.md)。

这份文档讲的是这一轮为什么继续做：

- `shell/` 模块
- 提示符
- 内建命令
- 命令注入自测
- 正常启动后的最小交互循环

---

## 1. 为什么“能读一整行”还不够

上一轮你已经有了：

- 控制台回显
- 退格
- `console_read_line()`

也就是说，
内核已经能拿到：

> “用户刚刚输入的一整行字符串”

但如果拿到了这一行以后，
系统还是什么都不做，
那它还不是 shell。

shell 最基础的意义就是：

> 读到一行输入之后，决定这行字到底代表什么操作。

所以这一轮的目标就是把流程从：

```text
键盘
-> 字符
-> 一整行
```

推进成：

```text
键盘
-> 字符
-> 一整行
-> 解析成命令
-> 执行并输出结果
```

---

## 2. 为什么单独拆 `shell/` 模块

如果继续把命令解析逻辑塞在 `kernel_main.cpp`，
很快就会出现两个问题：

1. 初始化代码和交互代码混在一起
2. 后面一旦命令多起来，主入口会越来越乱

所以这一轮专门新增了：

- `kernel/shell/shell.hpp`
- `kernel/shell/shell.cpp`

这意味着现在模块边界更清楚了：

- `keyboard/` 管“硬件输入进来”
- `console/` 管“字符怎么显示、怎么读成一整行”
- `shell/` 管“这一行到底要执行什么”

---

## 3. 这一轮 shell 到底支持什么

第一版 shell 故意只做 3 个内建命令：

1. `help`
2. `mem`
3. `ticks`

为什么只做这么少？

因为目标不是一口气做成完整终端，
而是先把最小主链打通：

- 提示符能出现
- 命令能读进来
- 命令能被识别
- 结果能被打印出来

这 4 点一旦通了，
后面你再继续加：

- `clear`
- `reboot`
- `echo`
- `ls`

结构就已经有地方挂了。

---

## 4. 这 3 个命令分别有什么意义

### 1. `help`

它不是“锦上添花”，
而是第一版 shell 最适合做的命令之一。

因为它能验证：

- shell 已经能识别字符串命令
- shell 已经能输出多行文字

### 2. `mem`

这个命令会打印：

- 当前还能分配多少页
- 大约还剩多少字节

它的意义是：

> shell 第一次真正把“内核内部状态”暴露成了可查询信息。

### 3. `ticks`

这个命令会打印当前全局时钟 tick。

它的意义是：

> shell 第一次能把“系统正在运行中的动态状态”查询出来。

所以这 3 个命令其实分别验证了三件不同的事：

- `help`：静态文本输出
- `mem`：内存子系统状态查询
- `ticks`：运行时动态状态查询

---

## 5. 为什么 shell 输出做成“字符回调”

这轮 `shell` 模块没有直接写死去操作 VGA 或串口，
而是只要求外界提供一个最小能力：

> 给我一个“输出 1 个字符”的回调

为什么这样设计？

因为 shell 自己其实不应该关心：

- 我是在写 VGA
- 还是在写串口
- 还是两边一起写

这一层分离以后，
`kernel_main.cpp` 就可以决定：

> shell 的每个字符同时写到控制台和串口

这样就带来两个好处：

1. 手工运行时，你能在屏幕上看到提示符和输出
2. 自动测试时，串口日志也能抓到同样内容

---

## 6. `shell_execute_line()` 在做什么

这个函数是 shell 真正的核心入口。

它的逻辑很朴素：

1. 如果是空行，就返回 `empty`
2. 如果是 `help`，执行 `help`
3. 如果是 `mem`，执行 `mem`
4. 如果是 `ticks`，执行 `ticks`
5. 否则打印 `unknown command: ...`

为什么要返回结果类型？

因为这样上层测试就能知道：

- 这条命令是成功执行了
- 还是只是空行
- 还是未知命令

这对于自动验证很有用。

---

## 7. 为什么还要专门测“未知命令”

如果你只测成功路径，
shell 看起来会很美好，
但一旦用户输错，
就不知道系统会不会乱掉。

所以这一轮自动测试里故意加了一条：

- `bad`

预期结果不是崩溃，
也不是静默无反应，
而是：

```text
unknown command: bad
```

这说明第一版 shell 已经具备最基础的错误反馈能力。

---

## 8. 这一轮自动化自测怎么做

这一轮不是手工在 QEMU 里敲命令，
而是继续沿用前面已经打通的输入注入链，
把下面这些命令一条条塞进去：

1. `help`
2. `mem`
3. `ticks`
4. `bad`

每一条都会走完整主链：

```text
注入扫描码
-> IRQ1
-> 字符缓冲区
-> console_read_line()
-> shell_execute_line()
-> shell 输出
```

这一步很重要，
因为它证明 shell 不是“直接伪造命令字符串去测解析器”，
而是真正复用了你前面几轮做出来的输入系统。

---

## 9. 正常启动后为什么要真的进交互循环

如果 shell 只做 smoke test，
那它还是更像“测试代码”，
不是“用户真的能用的东西”。

所以这一轮在正常启动路径里，
测试跑完以后还会继续进入：

- `shell_run_forever()`

它会反复做：

1. 打提示符 `os64> `
2. 读一行
3. 执行命令
4. 再回去等下一行

这一步的意义非常大，
因为它说明现在这个内核第一次已经有了真正的交互入口。

---

## 10. 日志里这一轮重点看什么

现在串口日志除了前面的输入测试外，
还应该继续看到：

- `shell_line=help`
- `commands:`
- `shell_line=mem`
- `mem_free_pages=...`
- `mem_free_bytes=...`
- `shell_line=ticks`
- `ticks_current=...`
- `shell_line=bad`
- `unknown command: bad`
- `shell ok`

这几组日志分别表示：

1. 命令行确实被正确读出来了
2. `help` 的多行输出成功了
3. `mem` 能访问页分配器状态
4. `ticks` 能访问时钟状态
5. 未知命令能得到明确错误反馈
6. 整条 shell 主链测试通过

---

## 11. 这一轮完成后，你真正多了什么能力

现在内核已经不只是：

- 会收到键盘输入
- 会回显字符
- 会读一整行

而是继续多了：

1. 真正的提示符
2. 第一版命令解释
3. 第一版内建命令
4. 正常启动后的持续交互循环

这意味着你的系统现在第一次开始像一个“真正可交互的操作系统内核原型”了。

---

## 12. 这一版还缺什么

现在它仍然只是最小 shell，
所以还有这些限制：

1. 还没有命令历史
2. 还没有 `clear`
3. 还没有自动补全
4. 还没有文件系统命令
5. 还没有进程/任务相关命令

这都正常，
因为这一轮目标只是：

> 先把“整行输入”升级成“可执行命令”。

---

## 13. 下一步最合理做什么

有了最小 shell 之后，
下一步最自然的通常就是：

1. 再加 2 到 3 个内建命令
2. 做 `clear`
3. 做 `echo`
4. 做最小内核监控命令，比如查看中断计数、键盘状态、堆状态

如果按“最适合继续学懂”的顺序，
我建议下一步优先做：

> 补几个更有观察价值的内建命令，比如 `heap`、`irq`、`clear`。

一句话总结这一轮：

> 现在内核不只是能读一整行输入，而是第一次真正能把这行输入当成命令执行了。
