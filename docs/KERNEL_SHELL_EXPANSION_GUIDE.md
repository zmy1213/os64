# 从最小 Shell 到可观察命令 + 带参数命令

这份文档讲的是：

- 为什么最小 shell 做完以后，下一步还是继续扩 shell，而不是立刻跳去文件系统
- 为什么先补 `heap` / `irq` / `clear`
- 为什么再继续补 `uptime` / `echo`
- 这一轮 shell 是怎么从“整行硬匹配”变成“命令名 + 参数”的

---

## 1. 为什么这一步还是继续做 shell

很多人第一次做到：

- 能进 64 位内核
- 能收键盘中断
- 能读一整行
- 能执行 `help` / `mem` / `ticks`

就会很想立刻跳去写：

- 文件系统
- 任务调度
- 用户态
- syscall

但这时候最合理的下一步通常并不是“功能跨度更大”，
而是先把已经有的交互入口继续打磨稳。

原因很简单：

> 你现在已经第一次有了“人可以直接和内核说话”的入口，最值钱的事情就是先把这个入口变得更有观察力。

所以这一步继续做 shell，
不是因为 shell 比文件系统“高级”，
而是因为它是你当前最省力、学习反馈最快的调试面板。

---

## 2. 为什么先补 `heap` / `irq` / `clear`

最小 shell 那一轮只有：

1. `help`
2. `mem`
3. `ticks`

这 3 个命令已经能证明：

- 字符串命令能被识别
- shell 能打印多行文本
- shell 能读内核状态

但如果你继续往前做内核，
你最常想看的状态通常不止这三个。

最自然的下一步就是把下面这三类信息暴露出来：

### 1. `heap`

它回答的是：

> “内核堆现在到底用了多少，还剩多少？”

因为你前面已经做了页分配器和内核堆，
如果没有一个能直接查的命令，
后面一旦堆分配行为不对，
你只能回去翻串口初始化日志。

### 2. `irq`

它回答的是：

> “时钟和键盘中断现在到底有没有继续在跑？”

这个命令把下面几类动态状态直接暴露出来：

- 当前 tick 数
- PIT 频率
- 键盘 IRQ 次数
- 字符缓冲区里还剩多少字符
- 缓冲区有没有丢字符

这对后面继续做交互、输入、甚至任务切换都非常关键。

### 3. `clear`

它回答的不是“内核内部状态”，
而是“交互界面是否能被整理”。

如果 shell 一直往下刷屏，
你后面连续试命令时很快就会看乱。

所以 `clear` 的意义是：

> shell 第一次不只是“显示信息”，而是开始真正管理自己的交互界面。

---

## 3. 为什么 `clear` 不直接让 shell 自己去操作 VGA

这里有一个很重要的模块边界问题。

`shell/` 的职责是：

- 识别命令
- 决定执行什么
- 把结果输出出去

`console/` 的职责才是：

- 光标在哪
- 哪一行该清空
- VGA 文本缓冲怎么写

所以 `clear` 这一轮没有让 `shell.cpp` 直接改 VGA 内存，
而是给 `ShellOutput` 多加了一个回调：

- `write_char`
- `clear`

这样 shell 只表达一件事：

> “用户执行了 clear，请把输出区域清掉。”

至于到底怎么清，
还是由控制台层决定。

这就是一个很典型的内核分层思路：

> 上层表达意图，下层负责碰硬件细节。

---

## 4. 为什么做完这三个后，还要继续补 `uptime` / `echo`

到这里 shell 虽然已经更有用了，
但它仍然有一个明显限制：

> 它还只会识别“整行刚好等于某个固定单词”。

这意味着它还不会理解：

- 命令名
- 命令后面跟的参数

如果你想让 shell 真正像一个命令行，
下一步最自然的就是第一次引入：

1. `uptime`
2. `echo <text>`

为什么是这两个？

### `uptime`

它适合做“继续查询内核运行状态”的命令。

它和 `ticks` 的区别是：

- `ticks` 只是原始计数器
- `uptime` 开始把原始 tick 解释成“系统已经跑了多久”

也就是说，
它第一次把“裸计数”升级成了“有意义的时间信息”。

### `echo <text>`

它适合做“第一条真正带参数的命令”。

因为它足够简单：

- 不改内核状态
- 不依赖复杂子系统
- 只需要把命令后的参数重新打印出来

但它又非常关键，
因为它证明了一件新的事：

> shell 已经不再只是查固定指令，而是开始理解“命令 + 参数”这种结构。

---

## 5. 这一轮解析逻辑到底升级了什么

以前的最小版本大概是这种思路：

```text
如果 line == "help"
  就执行 help

如果 line == "mem"
  就执行 mem
```

这叫“整行硬匹配”。

它适合第一版，
但一旦你想支持 `echo hello`，
问题就来了：

```text
整行已经不再只是命令名，
后面还会跟额外内容。
```

所以这一轮 shell 的思路升级成：

```text
先跳过前导空格
-> 识别第一个单词是不是命令名
-> 如果命令后面还有内容，把它当成参数区
-> 再决定执行哪个命令
```

你可以把它理解成：

> 以前 shell 把整行看成“一个整体字符串”，现在 shell 开始把这一行拆成“命令头 + 剩余参数”。

这一步非常重要，
因为后面如果你再加：

- `echo hello`
- `reboot now`
- `cat file.txt`
- `run task1`

都要靠这类最小解析能力。

---

## 6. `uptime` 命令到底做了什么

`uptime` 内部没有引入新的硬件，
它只是复用了你前面已经做好的定时器系统：

- `timer_tick_count()`
- `timer_frequency_hz()`

然后把它们拼起来算出：

- 当前 tick 数
- 当前定时器频率
- 按这个频率换算出的毫秒数

也就是说：

```text
uptime_ms = ticks * 1000 / frequency_hz
```

这很适合小白理解，
因为它说明：

> 所谓“系统运行时间”，并不是 CPU 凭空知道的，而是内核用周期性时钟中断一点点累计出来的。

---

## 7. `echo` 命令到底做了什么

`echo` 做的事很简单：

1. 找到命令名 `echo`
2. 把命令后面的剩余文本当作参数
3. 再把这段参数原样打印回去

它看起来很小，
但它其实是第一条真正证明“参数传递链”打通的命令。

你现在第一次有了这条链：

```text
键盘输入
-> 一整行字符
-> shell 识别命令名
-> shell 拿到后续参数
-> 内建命令消费这些参数
-> 输出结果
```

---

## 8. 自动测试为什么也要一起升级

这一轮不是只在代码里补两个命令就算完，
而是继续把命令注入自测一起扩掉。

现在 smoke test 会继续通过键盘扫描码注入下面这些命令：

1. `help`
2. `mem`
3. `ticks`
4. `heap`
5. `irq`
6. `echo hi42`
7. `uptime`
8. `clear`
9. `bad`

这件事很重要，
因为它说明新能力不是“单独调函数碰巧能跑”，
而是继续经过完整真实链路：

```text
扫描码
-> IRQ1
-> 字符缓冲
-> console_read_line()
-> shell_execute_line()
-> shell 输出
```

---

## 9. 这一轮完成后，你真正学到了什么

到这里你学到的已经不只是“做几个 shell 命令”。

你实际上在学 4 件很核心的内核工程思维：

1. 可观察性
   先把堆、中断、时间这些关键状态做成能随时查询的命令。
2. 分层
   `clear` 通过回调让 shell 表达意图，让 console 处理硬件细节。
3. 解析
   shell 开始从“整行相等判断”升级到“命令名 + 参数”的最小解析器。
4. 回归测试
   每补一条能力，都继续沿真实输入链路自动化验证。

---

## 10. 下一步最合理做什么

做到这里以后，
下一步最合理的通常不是立刻去碰文件系统，
而是先把交互体验再补一层。

最自然的两个方向是：

1. `history`
   让 shell 记住最近执行过的命令。
2. 更好的行编辑
   比如左右移动、删除键、简单的重新编辑。

如果按“最适合继续学懂”的顺序，
我更建议先做：

> `history`，因为它会让你第一次认真管理 shell 自己的状态，而不只是临时执行一行命令。

一句话总结这一轮：

> 现在这个内核的 shell，已经从“只会认固定单词”继续长成了“能查更多状态、还能吃参数”的第一版命令行。
