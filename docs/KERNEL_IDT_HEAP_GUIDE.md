# 从页表管理器到 IDT + 内核堆

这份文档专门讲这一轮为什么继续做：

- 最小 IDT
- 异常入口 stub
- page fault 基础处理
- 第一版内核堆

---

## 1. 为什么页表之后，最该补的是 IDT 和异常处理

上一轮你已经会做：

- `alloc_page()`
- `map_page()`
- 在一个新的虚拟地址上完成读写

这说明分页开始真的工作了。

但问题也跟着来了：

> 以后凡是“地址错了、页没映射、权限不对”，CPU 都不会温柔地提醒你，而是直接异常。

如果这时你还没有 IDT 和异常处理，
你看到的常常只会是：

- 黑屏
- 卡死
- QEMU 直接退出

所以这一步最重要的目标不是“做更多功能”，
而是先把：

- 出错时至少能看见“错在哪”

这条救命链搭起来。

---

## 2. 这一轮到底补了哪四块

### 第一块：IDT

IDT 你可以先理解成：

> CPU 发生异常或中断时，要去哪里找处理函数的目录表。

这一轮我们先只放最关键的几个槽位：

- 0：除零
- 6：非法指令
- 8：double fault
- 13：general protection fault
- 14：page fault

也就是说，
现在不是“所有异常都完善支持”，
而是先把最常见、最容易在内核早期遇到的几类异常接住。

### 第二块：异常入口 stub

CPU 发生异常时，并不是直接跳到一个普通 C++ 函数。

它会先：

1. 自动往当前栈里压一些现场
2. 跳到你在 IDT 里登记的入口地址

这个入口最适合先用汇编写，
因为它离 CPU 硬件行为最近。

所以这轮专门加了异常 stub，负责：

1. 补上统一的 `vector`
2. 对没有错误码的异常手工补 `error_code = 0`
3. 把 `CR2` 读出来
4. 调到 C++ 里的 `kernel_handle_exception()`

### 第三块：page fault 基础处理

page fault 是最重要的早期异常之一。

你可以先把它理解成：

> CPU 访问了一个当前页表不允许访问的地址。

比如：

- 这个虚拟地址根本没映射
- 这个页不可写，但你去写了
- 这个页不可执行，但你去执行了

这轮做的事不是“恢复执行”，
而是先做到：

- 把异常名字打出来
- 把异常向量号打出来
- 把错误码打出来
- 把 `CR2` 里的 fault address 打出来
- 把出错时的 `RIP` 打出来

这对小白非常重要，
因为你第一次终于能看到：

- “到底是哪一个地址炸了”
- “是读/写/权限哪一类问题”

### 第四块：第一版内核堆

前面你已经能拿到物理页，
但写内核时你不可能每次都手动想：

- 先拿物理页
- 再想虚拟地址
- 再手动 `map_page`

所以要补一个更顺手的接口：

- “我要一块连续内存”

这就是第一版内核堆的意义。

---

## 3. 第一版内核堆是怎么工作的

这一版故意保持得非常简单：

- 不支持 `free`
- 不做碎片整理
- 不做复杂元数据
- 只会一路往前分配

它本质上是一个：

> bump allocator

你可以把它理解成：

- `next_free` 指向“下一刀该从哪里切”
- 你要 64 字节，它就从这里切 64 字节给你
- 然后 `next_free` 往后推进

如果这次分配需要访问新的虚拟页，
它会自动：

1. 向底层页分配器申请物理页
2. 用 `map_page()` 把新的堆虚拟页接到页表里
3. 把新页清零
4. 再把内存返回给你

所以这一步真正打通的是：

```text
alloc_page()
-> map_page()
-> heap_alloc()
```

---

## 4. 为什么堆从 4 MiB 开始

这一轮堆虚拟地址起点选在：

- `0x400000`

也就是 4 MiB。

原因不是“标准规定必须这样”，
而是当前阶段这样最容易看懂：

1. 它明显高于 bootloader 当前只保证的低 2 MiB 恒等映射区
2. 这样一来，你能明确看到：
   - 这块内存不是“原来就能直接访问”
   - 而是内核自己新映射出来的
3. 地址也不算太高，调试输出很好认

同时这轮先把堆上界临时定在：

- `0x800000`

也就是 8 MiB。

这不是最终设计，
只是为了让第一版堆行为更可控。

## 5. 这一轮怎么测试

### 正常启动链测试

命令：

```bash
make test-stage1
```

这会验证：

- BIOS -> stage1 -> stage2 -> long mode -> kernel
- IDT 已初始化
- E820 和物理页分配器可用
- `map_page()` 可用
- 堆分配可用

通过时你会看到关键日志：

- `idt ok`
- `map_page ok`
- `heap init ok`
- `heap_small=0x...`
- `heap_large=0x...`
- `heap_cross_page_value=0xFEDCBA9876543210`
- `heap alloc ok`

其中：

- `heap_small` 证明小块分配已经能用
- `heap_large` 证明更大的块也能分配
- `heap_cross_page_value` 证明这块大分配已经跨页成功，不是只在单页里凑巧工作

### 专门的 page fault 冒烟测试

命令：

```bash
make test-page-fault
```

它会构建一个单独测试镜像，
在正常初始化和堆测试之后，
故意写一个没有映射的地址：

- `0x900000`

预期不是“继续成功运行”，
而是故意触发 page fault，
然后串口里应该能看到：

- `exception=page fault`
- `vector=0x000000000000000E`
- `fault_address=0x0000000000900000`
- `fault_rip=0x...`

这就说明：

- IDT 生效了
- page fault 入口 stub 生效了
- `CR2` 被正确读出来了
- C++ 异常处理函数真的接到了现场信息

---

## 6. 这一轮完成后，你真正多了什么能力

现在这个内核已经不只是：

- 能启动
- 能映射一个页

而是已经开始具备：

1. 出错时能报告异常
2. 能在高于 boot 恒等映射区的虚拟地址上分配内存
3. 能把页分配器、页表管理器、堆分配器串起来

这三个能力组合起来之后，
你后面再做：

- 更正式的内核堆
- 更完整的虚拟内存管理
- slab / buddy
- 用户态地址空间

才有真正继续扩的基础。

---

## 7. 这一版的局限也要看清楚

现在它还只是第一版，所以有这些明确限制：

1. 堆分配器只会往前长，不支持释放
2. 页表页目前仍然要求落在 bootloader 已映射的低 2 MiB 内
3. page fault 现在只会打印信息然后停机，不会恢复执行
4. 还没有真正接硬件中断控制器，也没有时钟中断和键盘中断

这些都不是 bug，
而是当前阶段故意不展开的复杂度。

---

## 8. 下一步最合理做什么

这一步之后，最自然的下一步通常是：

1. 做更正式的内核堆接口
2. 接管并扩展 bootloader 留下的临时页表
3. 开始做 IDT 的更多异常项和通用 trap 打印
4. 然后再进入 PIC/APIC、时钟中断、调度器骨架

一句话总结这一轮：

> 现在内核不只是“能映射内存”，而是开始具备“出错能看见、要内存能自己长”的能力。
