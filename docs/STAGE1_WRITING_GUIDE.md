# Stage1 写入说明

这份文档专门讲：

> 下一步为什么先写 `stage1 boot sector`，以及写它的时候到底在写什么。

注意，这里说的“写入过程”不是泛泛而谈，而是：

- 先创建哪些文件
- 每个文件承担什么职责
- 为什么按这个顺序写
- 每一小步怎么测

---

## 1. 为什么下一步先写 Stage1

如果你完全不用 `Limine`，那整个启动链最先执行的，不是 `kernel_main()`，也不是 `stage2`，而是 BIOS 读进来的第一个扇区。

也就是说：

```text
上电
-> BIOS
-> 把磁盘第一个扇区读到 0x7c00
-> 从 0x7c00 开始执行
```

所以你下一步最先写的真实代码，必须是：

- `boot/stage1.asm`

原因很简单：

- 没有它，BIOS 不会执行你的代码
- 没有它，`stage2` 没机会被加载
- 没有它，内核根本轮不到出场

这就是为什么我们不先写 `kernel/main.cpp`。

不是内核不重要，而是：

> 现在最先缺的不是“内核逻辑”，而是“让 CPU 先执行到你代码里的入口”。

---

## 2. 下一步一共要创建哪些文件

在真正开始写第一版时，最合理的是只创建这几样：

```text
os64/
├── boot/
│   └── stage1.asm
├── scripts/
│   └── build-stage1-image.sh
├── Makefile
└── disk.img            # 构建产物，不进仓库
```

### 为什么先只写这几个

因为第一步目标不是“进入 64 位内核”，而是：

> 先让 BIOS 执行你的 boot sector，并在屏幕上打印一句 `stage1 ok`

如果这一步都没通，后面任何更复杂的东西都没有意义。

所以第一轮不要创建：

- `stage2.asm`
- `kernel/main.cpp`
- `linker.ld`
- `memory.cpp`
- `sched.cpp`

这些后面都会有，但不是现在。

---

## 3. 第一轮最小目标是什么

第一轮不是加载 second stage。

第一轮只做一件事：

> 在 `QEMU` 里启动后，屏幕能出现 `stage1 ok`

### 为什么第一轮只做打印

因为 bootloader 调试非常脆弱。

如果你一上来就同时做：

- 初始化寄存器
- 打印
- 读磁盘
- 跳转 second stage

那么一旦黑屏，你很难知道死在哪。

所以要先把问题拆小。

第一轮先证明 3 件事：

1. `disk.img` 的第一个扇区是你写的
2. BIOS 确实跳进了你的代码
3. 你的代码确实能影响屏幕输出

只要这 3 点成立，后面再加读盘逻辑才不会盲飞。

---

## 4. Stage1 会写什么内容

第一版 `stage1.asm` 不应该太复杂。

它只应该包含下面这些部分：

### 1. 声明这是 16 位代码

你通常会写：

```asm
bits 16
org 0x7c00
```

### 为什么这么写

#### `bits 16`

因为 BIOS 把你带进来时，CPU 还在 16 位实模式。

这时候不是 32 位，也不是 64 位。

#### `org 0x7c00`

因为 BIOS 默认会把第一个启动扇区装载到内存地址 `0x7c00`。

这条指令的意思不是“把代码真的搬过去”，而是告诉汇编器：

> 你生成的这些地址，按“代码会在 `0x7c00` 执行”来计算。

如果这里不对，很多地址引用都会错。

---

### 2. 跳过数据区，进入主入口

你通常会先放一个短跳转：

```asm
jmp start
```

### 为什么这么写

因为 boot sector 里经常会混合：

- 代码
- 字符串
- 数据
- 启动签名

先跳到主入口，可以让布局更清楚。

---

### 3. 建立最小运行环境

你接下来通常会做这些事：

- 初始化 `DS`
- 初始化 `ES`
- 初始化 `SS`
- 设置 `SP`

### 为什么这么写

因为 BIOS 虽然把控制权交给你了，但很多寄存器值不能盲信。

尤其是：

- 数据段寄存器
- 栈段寄存器
- 栈指针

如果你不手动设，后面一旦：

- 调用子程序
- 压栈
- 取字符串地址

就可能直接出错。

### 为什么要设置栈

因为你后面至少会做：

- `call`
- `ret`
- 可能还会 `pusha/popa`

这些都依赖栈。

所以“先能用栈”是刚需，不是优化项。

---

### 4. 保存启动盘号

BIOS 进入 boot sector 时，会把启动盘号放在 `DL` 里。

你通常应该把它保存下来。

### 为什么

因为后面读 second stage 时，你还要知道：

- 是从哪个磁盘启动的

如果不保存，后面读盘时可能找错设备。

这一步现在第一轮即使还不用，也建议先记住这个设计。

---

### 5. 打印字符串

第一轮最推荐的方式是：

- 使用 BIOS 视频中断 `int 0x10`

### 为什么不用直接写 VGA 缓冲区

可以直接写，但第一阶段更推荐用 BIOS 打印，原因是：

- 你还在 16 位实模式
- BIOS 服务直接可用
- 代码短
- 调试直观

直接写 VGA 也能做，但 BIOS 打印在 stage1 更简单。

### 这一轮要打印什么

就一行：

```text
stage1 ok
```

### 为什么只打印这句

因为它代表三件事：

- 你的镜像是对的
- 你的启动扇区被执行了
- 你的最小执行环境已经够用了

---

### 6. 死循环停住

第一轮打印成功后，不要立刻继续做复杂逻辑。

直接停住就行。

可以是：

- `jmp $`
- 或 `cli` + `hlt`

### 为什么

因为第一轮目标只是验证“入口和打印是活的”。

停住之后，现象最稳定。

如果你打印完又继续跑，很可能很快跳飞，让你误以为打印也有问题。

---

### 7. 填充到 512 字节并写启动签名

最后必须补：

- 填充到 510 字节
- 最后两个字节写 `0x55` `0xaa`

### 为什么

因为 BIOS 判断“这个扇区是不是可启动”的经典条件之一，就是扇区末尾的启动签名。

没有这个签名，很多 BIOS 会直接认为：

- 这不是可启动扇区

### 为什么一定是 512 字节

因为传统 BIOS 启动的最小读取单位就是扇区，第一扇区就是 512 字节。

所以第一版 `stage1.bin` 必须满足：

- 大小正好 512 字节

---

## 5. 为什么第一轮先不写 second stage

这个问题很关键。

你当然最终要写 `stage2`，但不是第一轮。

### 原因 1：你现在最先要验证的是入口，不是加载器复杂逻辑

如果连 `stage1 ok` 都看不到，你根本不能证明：

- BIOS 真进来了
- 你的镜像布局是对的
- 你的扇区签名是对的

### 原因 2：读盘失败会让问题来源变复杂

如果你第一轮就加 `int 0x13` 读盘，一黑屏你就要同时怀疑：

- 扇区是不是没对齐
- 盘号是不是丢了
- CHS/LBA 参数是不是错了
- 读入地址是不是冲突
- 跳转地址是不是错了

这对小白很不友好。

### 原因 3：先把“最小启动回路”打通更稳

第一轮只做：

`BIOS -> stage1 -> print -> stop`

第二轮再做：

`BIOS -> stage1 -> load stage2 -> stage2 print -> stop`

这是最稳的顺序。

---

## 6. 第一轮写入顺序应该是什么

如果现在开始动手，真正合理的写入顺序是：

### Step A：创建 `boot/stage1.asm`

写最小 boot sector。

这一步只需要：

- 16 位声明
- 入口
- 设置段寄存器和栈
- 打印函数
- `stage1 ok`
- 启动签名

### Step B：创建 `scripts/build-stage1-image.sh`

它负责：

1. 用 `nasm` 汇编 `stage1.asm`
2. 检查输出是否是 512 字节
3. 创建最小 `disk.img`
4. 把 `stage1.bin` 写到第一个扇区

### 为什么先写脚本

因为以后你会反复改 `stage1.asm`。

如果每次都手敲：

- 汇编命令
- `dd`
- `truncate`
- `hexdump`

会很烦，也容易出错。

### Step C：手工验证镜像

先看：

- 大小
- 扇区尾部签名

### Step D：用 `QEMU` 启动

运行：

```bash
qemu-system-x86_64 -drive format=raw,file=disk.img
```

### Step E：观察现象

预期结果：

- 屏幕出现 `stage1 ok`
- 然后系统停住

只要这样，就算第一轮成功。

### 这次实现里为什么还加了串口输出

除了屏幕打印，这次实现还会把同样的字符串写到串口 `COM1`。

原因不是为了炫技，而是为了测试：

- 屏幕输出适合人眼看
- 串口输出适合终端自动抓取

这样你本地既可以打开 QEMU 窗口看 BIOS 屏幕，也可以在命令行下跑：

```bash
make run-stage1
```

然后直接在终端看到 `stage1 ok`。

如果想做更稳的自动验证，可以跑：

```bash
make test-stage1
```

它会：

1. 启动 QEMU
2. 把串口输出写到日志文件
3. 等待 2 秒
4. 主动结束 QEMU
5. 检查日志里有没有 `stage1 ok`

这样即使 QEMU 的 debug-exit 行为在当前环境里不稳定，自动测试仍然能做。

### 这次实现里为什么还加了 QEMU debug exit

如果 boot sector 只打印后死循环，自动化测试会卡住。

所以这次实现额外用了 QEMU 的 `isa-debug-exit` 设备：

- 打印完成后向一个特定 I/O 端口写值
- QEMU 收到后主动退出

这样做的原因不是因为真实硬件需要它，而是：

> 这样更适合开发阶段快速回归测试。

在真实硬件上，这个端口写出通常不会有意义；在 QEMU 里，它能让测试自动结束。

---

## 7. 第一轮完成后你实际实现了什么

如果第一轮成功，你实际实现的是：

### 实现 1：你亲手做出了一个合法 boot sector

这不是普通应用程序。

它是 BIOS 能认出来并执行的启动扇区。

### 实现 2：你证明了你的构建链是通的

包括：

- 汇编
- 生成镜像
- QEMU 启动

### 实现 3：你拿到了“从零启动链”的第一个可验证里程碑

也就是：

`BIOS -> 你的代码`

这一步一旦成立，后面再往上叠 second stage、保护模式、long mode 才是有根的。

---

## 8. 第一轮怎么测试才算严格

不要只看“好像屏幕亮了”。

更严格一点，测试应该包括：

### 测试 1：二进制大小

检查：

- `stage1.bin` 是否正好 512 字节

### 测试 2：启动签名

检查：

- 最后两个字节是不是 `55 aa`

### 测试 3：QEMU 屏幕输出

检查：

- 是否稳定打印 `stage1 ok`

### 测试 4：停机状态稳定

检查：

- 打印后不会乱跳乱码

这说明你的最小流程是稳的。

---

## 9. 你现在最容易写错的点

第一轮最常见错误就是这些：

### 1. 忘了 `org 0x7c00`

后果：

- 地址引用错

### 2. 忘了启动签名

后果：

- BIOS 不认这个扇区

### 3. 栈没设好

后果：

- `call` / `ret` 直接出问题

### 4. 输出后没有停住

后果：

- 很快跑飞，误判问题来源

### 5. stage1 写得太多

后果：

- 超过 512 字节

所以第一轮一定要克制。

---

## 10. 一句话总结

下一步写 `stage1`，不是因为它最酷，而是因为：

> 它是整个“从零启动链”里，第一个 BIOS 真会执行的文件。

所以第一轮真正要写的不是“大内核”，而是一个非常小、非常稳、非常可验证的 boot sector。
