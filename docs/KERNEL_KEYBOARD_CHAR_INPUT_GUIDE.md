# 从键盘 IRQ 到最小字符输入

这份文档讲的是这一轮为什么继续做：

- 键盘字符环形缓冲区
- 扫描码到字符的第一版翻译
- `keyboard_try_read_char()`
- 自动化字符输入自测

---

## 1. 为什么“只会收扫描码”还不够

上一轮你已经有了：

- IRQ1 能进内核
- 内核能从 `0x60` 端口读出扫描码
- 能知道“最近一次按键硬件送来的编号是什么”

这说明：

> 键盘这条硬件中断链已经通了。

但这还不是“真正可用的输入系统”。

因为扫描码像 `0x1E`、`0x30` 这种值，
对硬件来说有意义，
对上层代码却不够友好。

上层以后真正想问的是：

- 有没有字符输入？
- 下一个字符是不是 `a`？
- 用户刚才是不是按了回车？

所以这一轮的目标就是把流程继续往前推一层：

```text
IRQ1 到来
-> 读出扫描码
-> 翻译成最小字符集
-> 放进字符缓冲区
-> 让内核代码按顺序读字符
```

---

## 2. 为什么要先做“环形缓冲区”

小白版可以这样理解：

如果你每次只记“最后一个字符”，
那用户连续按：

- `a`
- `b`
- `1`

最后你可能只剩下：

- `1`

前两个字符就丢了。

这显然不行。

所以要先准备一个小队列。

这一轮选的是：

> 环形缓冲区（ring buffer）

它的意思不是“很高级”，
而是：

1. 预留一段固定大小的数组
2. 新字符从 `write_index` 位置写进去
3. 读字符从 `read_index` 位置取出来
4. 指针走到数组末尾后，再绕回开头继续用

这样最适合做第一版键盘输入队列，
因为它：

- 很简单
- 不需要动态内存
- 非常适合 IRQ 里快速写入

---

## 3. 这一轮在 `keyboard.cpp` 里新增了什么

### 1. 字符缓冲区本体

现在模块里多了：

- `g_keyboard_char_buffer`
- `g_keyboard_char_read_index`
- `g_keyboard_char_write_index`
- `g_keyboard_char_count`

你可以把它想成：

- 一个小数组
- 一个“下次从哪里读”的指针
- 一个“下次往哪里写”的指针
- 一个“里面现在有多少字符”的计数器

### 2. 丢字符计数器

还多了：

- `g_keyboard_dropped_char_count`

为什么这东西有意义？

因为如果以后键盘输得很快，
而上层代码读得很慢，
缓冲区就可能满。

第一版最稳的做法不是偷偷覆盖旧数据，
而是：

> 先丢掉新字符，并明确记一笔“这次丢了多少个”。

这样测试时就能知道有没有悄悄出问题。

### 3. 扫描码翻译函数

现在不是每个扫描码都原样往外暴露了，
而是先经过一个最小翻译函数：

- 数字键 `0-9`
- 字母键 `a-z`
- 空格
- 回车
- 退格

这一步只支持最小字符集，
还没有做：

- Shift
- Caps Lock
- 符号键组合
- 中文输入

这很正常，
因为这一轮目标只是：

> 先把“硬件按键”变成“最小字符输入”。

---

## 4. 为什么要忽略 break code

键盘不是只会发“按下”，
还会发“松开”。

例如这一轮测试里：

- `0x1E` 是 `A` 按下
- `0x9E` 是 `A` 松开

如果你把这两个都翻译成字符，
那按一次 `A` 就可能在缓冲区里看到两个 `a`，
这当然不对。

所以这一轮先采用最简单规则：

> 只处理 make code（按下），忽略 break code（松开）。

在经典 Set 1 里，
一个很常见的经验规则就是：

> 最高位是 `1` 的很多扫描码，表示松开事件。

所以代码里先做了这个最小过滤。

---

## 5. `keyboard_try_read_char()` 到底在做什么

这个接口是这一轮最重要的用户态视角 API：

```cpp
bool keyboard_try_read_char(char* out_char);
```

它的意思是：

- 如果缓冲区里现在有字符
  - 取出 1 个
  - 写到 `out_char`
  - 返回 `true`
- 如果缓冲区里现在没有字符
  - 什么都不取
  - 返回 `false`

为什么先做 `try_read`，而不是一上来就做阻塞版 `read`？

因为 `try_read` 最容易验证，也最通用：

- 你可以轮询它
- 也可以以后把它包成阻塞接口
- 还可以以后给 shell、控制台、命令行复用

也就是说：

> 这一轮先把“取字符”这件事做成最小、最干净的基础接口。

---

## 6. 为什么 `keyboard_try_read_char()` 里要临时关中断

这里有一个非常真实的并发点：

- 普通内核代码可能正在读字符
- 同时新的键盘 IRQ 也可能突然到来

而 IRQ 处理函数会改：

- `write_index`
- `char_count`

读字符函数会改：

- `read_index`
- `char_count`

如果两边刚好同时改这些状态，
就可能把缓冲区搞乱。

所以这一轮的做法是：

1. 先保存当前中断标志
2. 临时 `cli`
3. 原子地完成一次“看有没有字符 + 取走一个字符”
4. 最后把原来的中断状态恢复

为什么这样做？

因为现在还是单核、最小内核，
这种“短暂关中断保护临界区”的做法最直接、最适合教学。

---

## 7. `handle_keyboard_irq()` 现在比上一轮多做了什么

上一轮它只做两件事：

1. 读 1 个扫描码
2. 记到 `g_keyboard_last_scancode`

这一轮多了后半段：

1. 尝试把扫描码翻译成字符
2. 如果翻译成功，就入队到字符缓冲区
3. 如果是 break code 或不支持的键，就忽略

所以它现在已经不是“只会记日志”的中断处理函数了，
而是开始真正生产上层可用输入。

---

## 8. `kernel_main()` 里的新自测在验证什么

这一轮自测不再只注入 1 个扫描码，
而是注入一整串：

1. `0x1E` -> `a`
2. `0x9E` -> `A` 松开，应该忽略
3. `0x30` -> `b`
4. `0x02` -> `1`
5. `0x39` -> 空格
6. `0x1C` -> 回车
7. `0x0E` -> 退格

这组数据为什么选成这样？

因为它能一次验证 4 件事：

1. IRQ 连续到来时，计数是不是正常增加
2. break code 会不会被错误地翻译成字符
3. 不同类型的字符能不能都进缓冲区
4. 读出来的顺序是不是 FIFO

这比只测一个 `A` 强很多，
因为它已经开始像真正的输入队列测试了。

---

## 9. 预期日志现在会多什么

这一轮通过后，
除了上一轮那些：

- `keyboard init ok`
- `keyboard irq1 enabled`
- `keyboard_irq_count=...`

现在还应该继续看到：

- `keyboard_char_count=6`
- `keyboard_char_0=0x0000000000000061`
- `keyboard_char_1=0x0000000000000062`
- `keyboard_char_2=0x0000000000000031`
- `keyboard_char_3=0x0000000000000020`
- `keyboard_char_4=0x000000000000000A`
- `keyboard_char_5=0x0000000000000008`
- `keyboard_buffer_remaining=0`
- `keyboard_dropped_chars=0`
- `keyboard ok`

这些日志分别说明：

1. 缓冲区里一共收到了 6 个有效字符
2. 第 0 个字符是 `a`
3. 第 1 个字符是 `b`
4. 第 2 个字符是 `1`
5. 第 3 个字符是空格
6. 第 4 个字符是回车
7. 第 5 个字符是退格
8. 读完以后缓冲区已经被清空
9. 这一轮没有因为缓冲区太小而丢字符

## 10. 怎么测试这一轮

命令还是这 3 个：

```bash
make test-stage1
make test-invalid-opcode
make test-page-fault
```

为什么后两个也要继续跑？

因为这轮虽然主要改的是键盘输入，
但它也改到了：

- IRQ 路径里的键盘行为
- 内核自测顺序
- 串口日志断言

所以还要确认：

> 新增字符缓冲区以后，原来的异常链并没有被顺手搞坏。

---

## 11. 这一轮完成后，你真正多了什么能力

现在这个内核已经不只是：

- 能收到一次键盘 IRQ
- 能记住“最后一个扫描码”

而是继续多了：

1. 连续输入不会只剩最后一个按键
2. 上层代码第一次能按顺序读字符
3. 输入开始从“硬件编号”提升成“最小文本流”

这一步很关键，
因为很多后续模块都依赖它：

- 控制台回显
- 最小 `getc()`
- 命令行
- shell

---

## 12. 这一版仍然有哪些限制

现在它仍然只是“最小字符输入”版本，
所以还有这些限制：

1. 还没有 Shift / Caps Lock
2. 还没有符号键映射，比如 `!`、`@`
3. 还没有方向键、功能键、扩展键处理
4. 还没有行编辑逻辑
5. 还没有真正显示“用户输入的字符”

这都正常，
因为这一轮目标只是：

> 先把“键盘中断”升级成“内核里真的能读字符”。

---

## 13. 下一步最合理做什么

有了字符缓冲区之后，
下一步最自然的通常是：

1. 做最小控制台回显
2. 做 `getc()` / `readline()` 风格接口
3. 处理退格对屏幕显示的影响
4. 做最小命令行

如果按“最适合继续学懂”的顺序，
我建议下一步优先做：

> 控制台回显 + 最小行输入。

一句话总结这一轮：

> 现在内核不只是知道“键盘来过了”，而是第一次真正能把输入按字符顺序读出来了。
